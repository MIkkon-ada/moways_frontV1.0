# 博维AI升级项目驾驶舱 — 前端交互设计工作流程

> 文档范围：`app/static/app.js` + `app/static/style.css`
> 技术栈：纯原生 JavaScript，无框架，单页应用（SPA）
> 最后更新：2026-05-24

---

## 一、整体架构

### 1.1 页面结构

系统共 7 个功能页面，通过顶部导航 Tab 切换，每次切换重新渲染当前页内容：

| 页面 ID | 中文名 | 核心作用 |
|---------|--------|---------|
| `dashboard` | 驾驶舱 | 全局状态总览，管理者视角 |
| `updates` | 提交进度更新 | 语音/会议/文字三种方式提交进度 |
| `confirmations` | AI确认中心 | 负责人审核 AI 建议后写入数据库 |
| `tasks` | 工作推进表 | 任务台账，按专项分组 |
| `achievements` | 成果库 | 阶段性成果管理 |
| `issues` | 问题与决策 | 问题跟踪 + 决策事项 |
| `people` | 组织与分工 | 专项与人员双视角 |

### 1.2 应用初始化流程

```
页面加载
  ↓
initRoleSwitch()        — 渲染角色切换下拉，默认选中"冯海林"，绑定切换事件
  ↓
initTabs()              — 渲染导航 Tab，绑定页面切换
  ↓
setupOrgInteractions()  — 给"组织与分工"页预绑定事件委托（点击/悬停）
  ↓
loadDynamicOrgData()    — 并行从 API 拉取专项和人员数据，更新 projectAreas / members
  ↓
updateRoleSwitchOptions() — 用 API 数据刷新角色下拉选项
  ↓
loadPage("dashboard")   — 渲染首页驾驶舱
```

**说明：** `initRoleSwitch()` 必须在 `loadDynamicOrgData()` 之前执行。原因：`loadDynamicOrgData()` 内部调用 `api()`，而 `api()` 需要读取 roleSwitch 的值来设置 `X-Current-User` 请求头。先初始化下拉保证身份识别正常。

## 二、用户身份与权限体系

### 2.1 身份识别方式

右上角有一个**角色切换下拉**（roleSwitch），选中谁就以谁的身份访问。每次 API 请求自动在 Header 里带上：

```
X-Current-User: <URL编码的用户名>
```

后端据此做权限过滤，返回该用户有权看的数据。

### 2.2 前端权限上下文

每个页面渲染时调用 `getCurrentUserContext()` 获取当前用户的权限信息：

```js
{
  name,                // 当前用户名
  isCEO,               // 冯海林
  isTechAdmin,         // 吴肖、郭熠彬
  isProcessGuard,      // 袁金玉
  isCoordinator,       // 刘万超、邹奇敏
  ownedProjects,       // 本人负责的专项
  coordinatedProjects, // 本人统筹的专项
  collaboratedProjects,// 本人协同的专项
  visibleProjects,     // 本人可见的专项列表
  canViewAll,          // CEO / tech_admin / process_guard
  canMaintainAll,      // CEO / tech_admin（可写入）
  canAssignAll,        // CEO / tech_admin / process_guard（可分配责任人）
}
```

### 2.3 权限对 UI 的影响

| 场景 | 影响 |
|------|------|
| 专项筛选下拉 | 只显示 `visibleProjects` 里的专项 |
| 工作推进表 | 只展示有权限专项的任务 |
| 成果库 | 只展示有权限专项的成果 |
| AI确认中心详情 | 有 `canMaintainAll` 或本专项负责人才能看到"确认写入"按钮 |
| 待分配队列 | 只有 `canAssignAll` 的人才能执行分配操作 |

> **设计原则：** 权限过滤在后端做。前端的 `visibleProjects` 仅用于控制 UI 选项范围，不做数据安全保障。

## 三、全局状态管理（state）

所有页面共用一个 `state` 对象，存储各页面的筛选条件、选中状态等：

```js
state = {
  // 进度更新相关
  previewPayload: null,        // 当前正在预览的提交表单数据
  previewSuggestion: null,     // AI 提取的建议结果
  updateInputMode: "voice",    // 输入模式：voice / meeting / text
  isRecording: false,
  recordingSeconds: 0,
  recordingTimer: null,

  // 确认中心
  selectedConfirmationId: null,
  confirmationTab: "待确认",   // 待确认 / 待分配 / 已确认

  // 驾驶舱筛选
  dashboardFilters: { project, owner, status, month },

  // 工作推进表筛选
  taskFilters: { project, status, owner, month },

  // 成果库筛选
  achievementFilters: { project, type, reuse, owner },

  // 问题与决策
  issueTab: "all",             // all / problem / decision / closed
  issueFilters: { priority, project, owner, status },

  // 组织与分工
  orgViewMode: "project",      // project / member
  selectedOrgProjectId: "",
  selectedOrgMember: "",
  hoveredOrgProjectId: "",
  hoveredOrgMember: "",
}
```

**状态变更规则：** 任何筛选条件改变 → 更新 state → 重新调用对应 `render*()` 函数，整页重渲染（无虚拟 DOM diff）。

## 四、核心数据流：进度更新闭环

这是系统最重要的工作流，贯穿三个页面。

```
[提交进度更新] → [AI确认中心] → [写入数据库]
      ↓                ↓                ↓
  录入内容         负责人审核        任务/成果/问题表更新
```

### 4.1 提交进度更新页（updates）

**三种输入模式，共享同一套 AI 提取流程：**

#### 语音更新模式
1. 用户点击录音按钮 → `toggleRecording()` 启动计时器，界面显示录音时长
2. 再次点击停止录音
3. 也可上传 .mp3 / .m4a / .wav 文件
4. 点击"生成 AI 提取建议" → `extractUpdate()` 读取表单，POST 到 `/api/updates/extract`
5. API 返回 AI 建议 → `renderSuggestionPanel()` 渲染右侧结果面板

#### 会议纪要模式
1. 粘贴会议纪要文本
2. 袁金玉点击"完成校对" → `markMeetingReviewed()` → 解锁"发布"按钮
3. 发布后进入 AI 提取流程，同上

#### 文字粘贴模式
1. 直接粘贴个人进度说明（豆包转写文本等）
2. 点击生成，进入 AI 提取流程

#### 辅助提示：四个引导问题
页面左侧始终显示四个结构化引导问题，帮助用户组织语言：
1. 本周完成了什么？
2. 形成了什么成果？（文件、工具、机制）
3. 当前有什么问题或卡点？需要谁协调？
4. 下周计划做什么？

### 4.2 AI 提取结果面板

`renderSuggestionPanel(suggestion)` 渲染以下字段：

| 字段 | 置信度 | 交互方式 |
|------|--------|---------|
| 识别专项 | 高 | `<select>` 下拉（含全部专项，AI识别到陌生项目时显示"新增"紫色标签） |
| 关联任务 | 中 | `contenteditable` 可直接编辑 |
| 完成事项 | 高 | `contenteditable`，多项用"；"分隔 |
| 形成成果 | 中 | `contenteditable` |
| 下周计划 | 中 | `contenteditable` |
| 需协调人 | 高/低 | `contenteditable` |

**置信度徽章含义：**
- 高（绿色）— AI 非常确定
- 中（橙色）— 需要负责人复核
- 低（红色）— AI 未能识别，需手动补充

**识别到问题时：** 如果 AI 提取到"当前风险/卡点"，自动在面板下方显示"自动创建问题卡"预览。

**三个操作按钮：**
- **提交确认中心** — `submitPreview()`，读取所有编辑后的字段，POST 到 `/api/updates`，跳转到确认中心
- **重新生成** — 重新调用 `extractUpdate()`
- **放弃** — `discardPreview()`，清空预览状态，恢复等待状态

### 4.3 AI 确认中心（confirmations）

**三栏式布局：左侧列表 + 右侧详情**

#### 三个 Tab 队列

| Tab | 含义 | 谁能操作 |
|-----|------|---------|
| 待确认 | 等待负责人校对写入 | 本专项负责人 / CEO / tech_admin |
| 待分配 | AI 未识别责任人，需先指定 | 袁金玉 / CEO / tech_admin（canAssignAll） |
| 已确认 | 已处理归档，只读 | 所有人可查看 |

#### 待确认详情的操作流程

```
点击列表中的记录
  ↓
openConfirmation(id) — 拉取详情，渲染右侧面板
  ↓
查看转写原文（可展开折叠）
查看 AI 提取字段（可 contenteditable 直接修改）
  ↓
三个操作按钮（权限控制显示）：
  ├── 退回重录 → rejectConfirmation()，填写退回原因，状态改为"需重录"
  ├── 转交袁金玉 → transferToYuanjinyu()，移入待分配队列
  └── 确认写入 → confirmWrite()，写入任务/成果/问题表，归入已确认
```

#### 待分配详情的操作流程

```
打开记录
  ↓
袁金玉从下拉选择责任人
  ↓
confirmAssign(id) — 指定责任人后移回待确认队列
  ↓
对应负责人在待确认队列看到该记录并完成确认
```

## 五、工作推进表（tasks）

### 5.1 页面结构

```
页头（标题 + 三个快捷入口按钮）
  ├── 语音更新 → 跳转 updates 页
  ├── 会议纪要导入 → 跳转 updates 页
  └── 人工确认更新 → 跳转 confirmations 页

筛选栏（4个下拉）
  ├── 专项（按权限过滤，只显示有权限的专项）
  ├── 状态（未启动/进行中/已完成/延期/暂缓）
  ├── 负责人
  └── 计划月份

任务分组列表（按专项分组，每组有状态统计徽章）
  └── 任务卡片（可展开 `<details>` 元素）
```

### 5.2 任务卡片交互

**折叠状态（summary 部分）：**
- 任务名称
- 负责人 / 统筹人 / 协同成员
- 计划时间（延期任务显示"已超期"红色警告）
- 状态徽章
- 新增任务显示"新增"标签（source_type 不含"Excel导入"的任务）

**展开后（detail 部分）：**
- 关键成果（文件链接自动识别为可点击链接）
- 完成标准
- 当前问题与需协调事项（有内容时显示，附"升级为问题/决策项"按钮）
- 操作日志（懒加载，展开时才请求 `/api/logs?target_type=task&target_id=xxx`）

**操作日志显示格式：**
```
2026-05-10 09:30  吴肖    新建任务
2026-05-15 14:20  许明良  状态：推进中 → 延期      ← 状态变更高亮
2026-05-22 11:00  许明良  状态：延期 → 已完成      ← 原状态删除线灰色，新状态蓝色加粗
2026-05-22 11:05  吴肖    修改：计划时间、问题备注
```

日志使用 `data-loaded` 属性防止重复请求：DOM 重渲染后属性消失，下次展开自动重新拉取。

### 5.3 任务排序规则

- 后端按 `created_at.asc()`（提交时间升序）返回
- Excel 导入任务和手工新增任务地位平等，都按提交时间排
- 前端不做二次排序，保留后端顺序

### 5.4 状态值映射

| 展示状态 | 实际可能的状态值 |
|---------|----------------|
| 进行中 | "推进中" / "进行中" |
| 延期 | "延期" / "风险" |
| 未启动 | "未开始" / "未启动" |

## 六、成果库（achievements）

### 6.1 页面结构

```
页头（标题 + 五项统计指标）
  ├── 成果总数（计划）
  ├── 计划中
  ├── 已形成
  ├── 已归档/可复用
  └── 归档提醒（未关联任务、缺版本号、需脱敏的成果数）

筛选栏（4个下拉）
  ├── 专项（按权限过滤）
  ├── 成果类型
  ├── 复用场景
  └── 负责人

成果分组（按专项分组）
  └── 每组一张横向表格
```

### 6.2 成果表格列结构

| 成果名称 | 类型 | 复用场景 | 负责人 | 版本 | 状态 | 文件 |
|---------|------|---------|--------|------|------|------|

成果名称列支持悬停显示完整名称（`title` 属性），名称过长时省略号截断。

### 6.3 归档警告规则

每条成果自动检查以下三项，有问题时在名称下方显示橙色警告标签：

| 检查项 | 条件 |
|--------|------|
| 需关联任务 | `related_task_id` 为空 |
| 重要成果需标记版本 | `version` 为空 且状态不是"计划中" |
| 需脱敏 | `reuse_tag` 为"客户交付" 或名称/类型含"案例包"/"客户验证记录" |

### 6.4 成果状态分类

| 状态值 | CSS类 | 含义 |
|--------|-------|------|
| 已归档 / 可复用 | `archived` | 已完成归档 |
| 已形成 / 已完成 | `formed` | 已形成但未归档 |
| 计划中（其他） | `planned` | 规划阶段 |

### 6.5 文件操作

"文件"列按钮：
- 有链接 → 显示"查看"
- 无链接 → 显示"待上传"，点击提示补充链接（当前为 toast，待完善为编辑弹窗）

## 七、问题与决策（issues）

### 7.1 页面结构

```
Tab 筛选（全部 / 问题·风险 / 待决策 / 已关闭）

筛选栏（优先级 / 专项 / 责任人 / 状态）

问题/风险区块
  └── 问题记录卡片列表

待冯海林决策区块
  └── 决策记录卡片列表
```

### 7.2 记录类型与编号规则

| 类型 | 编号前缀 | 触发条件 |
|------|---------|---------|
| 问题/卡点/风险 | `#P-xx` | `need_decision_by` 为空 |
| 决策项 | `#D-xx` | `need_decision_by` 有值 |

### 7.3 记录卡片内容

每张卡片显示：
- 编号（#P-01 格式）
- 问题描述
- 责任人 / 协同人 / 所属专项 / 预计解决时间
- 类型标签（风险/卡点/决策）
- 优先级标签（高/中/低）
- 状态标签
- "查看/处理"按钮
- 如有结论：展示结论文本

### 7.4 处理流程（弹窗）

点击"查看/处理" → `openIssueDetail()` → 打开 `openModal()` 弹窗：

```
弹窗内容：
  ├── 问题描述（只读）
  ├── 所属专项（只读）
  ├── 责任人（只读）
  ├── 优先级（可修改下拉）
  ├── 当前状态（可修改下拉）
  ├── 预计解决时间（可填写）
  └── 处理结论 / 决策结论（可填写，大文本框）

操作按钮：
  ├── 取消
  └── 保存 → saveIssueDetail()
        ├── 如状态有变更 → PATCH /api/issues/{id}/status
        └── PUT /api/issues/{id}（更新全部字段）
        └── 状态为"已关闭/已决策/已解决"时，自动切换到"已关闭"Tab
```

## 八、组织与分工（people）

### 8.1 双视角模式

通过右上角分段控件切换：

| 模式 | 交互逻辑 |
|------|---------|
| **专项视角** | 点击专项卡片 → 高亮所有相关成员，其余人员变暗 |
| **人员视角** | 点击人员卡片 → 高亮该人员参与的所有专项，其余变暗 |

两种模式下鼠标悬停均有即时预览效果（不需要点击）。

### 8.2 专项卡片

每张专项卡片显示：
- 专项名称
- 统筹人 / 负责人
- 进度条（从真实任务数据实时计算：已完成任务数 / 总任务数）
- 激活状态下显示角色标签（统筹/负责/协同）

### 8.3 人员卡片

分两排展示：
- **核心成员**（冯海林、刘万超、邹奇敏）— 大卡片，显示完整职责
- **执行成员** — 紧凑卡片，显示负责专项简称

每张卡片的角色标签：
- 无专项选中 → 显示 `member.role`（静态角色名）
- 选中某专项 → 显示该人在该专项的角色（统筹/负责/协同）

### 8.4 事件绑定策略

组织页使用**事件委托**，在页面容器上绑定一次 click/mouseover/mouseout，通过 `.closest()` 判断实际点击目标。原因：组织页内容会随角色切换反复重渲染，直接绑在子元素上会丢失。使用 `data-orgBound` 标志防止重复绑定。

## 九、驾驶舱（dashboard）

### 9.1 数据聚合

驾驶舱同时拉取四个接口的数据后再渲染：

```js
Promise.all([/api/tasks, /api/achievements, /api/issues, /api/confirmations/pending])
```

### 9.2 五个区块

#### 今日需关注（顶部告警栏）
- 待决策数 / 延期任务数 / 高风险数，点击直接跳转对应页面

#### 需要您处理（双栏）
- 待您决策：取 `need_decision_by === "冯海林"` 的问题，或 AI 确认中心的待处理条目（前三条）
- 主要风险：开放状态的问题/风险（前三条）

#### 任务状态总览
- 6个指标卡片：总任务数 / 延期 / 暂缓 / 进行中 / 已完成 / 未启动

#### 五个专项进度
- 每个专项一张进度卡片，显示完成率进度条和各状态数量
- 支持按专项/月份/状态/负责人筛选，筛选后只显示对应专项

#### 底部双栏
- 本周重点：未完成且未暂缓的任务，取前4条
- 已形成成果：最新4条成果

### 9.3 筛选交互

```
setDashboardFilter(name, value)
  → 更新 state.dashboardFilters
  → 重新调用 renderDashboard()
  → 整页重渲染（包括五个专项进度区块）
```

## 十、通用交互组件

### 10.1 模态弹窗

```js
openModal(htmlContent)   // 打开
closeModal()             // 关闭
```

用于问题详情、链接上传等需要临时焦点的操作。

### 10.2 Toast 提示

```js
toast("提示文字")
```

轻量通知，2.6 秒后自动消失，不打断操作流程。

### 10.3 空状态占位

```js
emptyState("暂无任务", "触发条件说明文字")
```

所有列表在数据为空时统一显示空状态，附带引导文字。

### 10.4 徽章组件

```js
badge(status)   // 根据状态值返回带颜色的 <span> 标签
```

状态颜色统一管理，不在各页面单独硬编码。

### 10.5 进度条组件

```js
progress(rate)  // 0-100 的数字，返回带颜色的进度条 HTML
```

颜色区间：< 30% 红色，30-70% 橙色，> 70% 绿色。

## 十一、数据规范化

### 11.1 专项名称标准化

```js
normalizeProject(name)
```

处理历史数据中的异名：如"风险"状态错误写入专项字段等边缘情况，统一映射到标准专项名。

### 11.2 问题记录标准化

normalizeIssueRecord(row)

## 十二、关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 专项筛选 | `<select>` 下拉 | 专项名称长，按钮 Tab 显示不全 |
| 任务排序 | `created_at.asc()` | Excel 是大纲参照物，提交时间反映真实工作顺序 |
| 操作日志 | 展开时懒加载 | 避免列表页一次性请求所有任务的日志 |
| 权限过滤 | 后端做，前端仅控制选项范围 | 双重过滤会掩盖后端权限失效的问题 |
| 组织页事件 | 事件委托 | 内容频繁重渲染，直接绑定会丢失 |
| 动态 org 数据 | API 拉取后更新静态 fallback | API 失败时保有静态数据，不白屏 |
| PROJECTS 变量 | `Object.defineProperty` getter | 从 `projectAreas` 动态派生，随 API 更新自动生效 |
