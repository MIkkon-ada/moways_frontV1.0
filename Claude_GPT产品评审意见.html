执行摘要 EXECUTIVE SUMMARY

本次评审对 bowei_ai_dashboard V0.2-alpha 的全部 Python 源代码进行了逐文件审阅，共识别 13 项工程缺陷，涵盖严重（Critical）、高（High）、中（Medium）、信息（Info）四个等级。

整体判断：GPT 在架构层面给出了合理的技术选型（FastAPI + SQLite + 前后端分离），路由分层、模型设计、日志模型均有基础工程意识。但在 AI 能力实现、安全边界、数据完整性、工程规范 四个维度上存在显著缺陷，其中"伪 AI 实现"和"零访问控制"两个问题的性质最为严重——前者存在对甲方的误导性描述，后者在任何联网场景下均构成安全漏洞。

该代码可作为本地演示脚手架，不应作为 MVP 验收基准，不建议未经修改直接进入联调或部署阶段。

1严重缺陷 · CRITICAL
CRITICAL C-01 AI 提取能力完全虚构，置信度指标造假

产品描述的核心卖点是"豆包转写文本 → AI 结构化提取"，但实际实现是一组正则关键词匹配规则，与任何大模型无关。更严重的是，代码向前端返回了一个虚假的 confidence 置信度分数，其计算逻辑是用字符数判断质量，而非真实的模型置信度。

app/services/extractor.py · 第 5, 71–73, 139 行
USE_LLM = False

def extract_update(...) -> dict:
    if USE_LLM:
        # Extension point: call an external LLM here...
        pass          ← 这里是空的，分支完全无效

    ...
    "confidence": 0.82 if len(text) > 40 else 0.58,   ← 用字数伪造置信度

USE_LLM = True 分支体是一个孤立的 pass 语句，执行后代码会继续运行下面的规则逻辑，开关对行为没有任何影响。这意味着即便未来有人将标志位改为 True，系统仍然不会调用任何 LLM，且不会报错，会静默返回错误结果。

前端展示的"AI 建议"和"置信度 82%"属于误导性描述。若产品演示以此为基础向甲方展示 AI 能力，构成功能性虚假陈述。

修复建议

在 extractor.py 中将 LLM 分支实现为真实调用（如 Anthropic API / OpenAI API），统一返回结构；或在所有界面中将当前功能明确标注为"规则模拟（Demo）"，不展示置信度数值。

CRITICAL C-02 系统无任何认证与授权机制

所有 API 端点均无鉴权保护。任何能访问服务地址的人，无需登录即可执行以下操作：读取全部任务、成果、人员信息；创建、修改、删除任意业务数据；确认或拒绝所有 AI 建议；获取完整的操作日志。

app/routers/tasks.py · 全部路由
@router.delete("/{row_id}")
def delete_task(row_id: int, db: Session = Depends(get_db)):
    # ← 无任何身份验证，任何人均可删除任意任务
    row = db.get(models.Task, row_id)
    db.delete(row)
    db.commit()
    return {"ok": True}
app/routers/confirmations.py · 第 88 行
row.confirmed_by = payload.operator   ← 确认人由请求方自填，无法验证身份

confirmed_by、operator 字段由请求体自行传入，意味着操作日志中的"操作人"可以被任意伪造。审计链路完全失去可信度。

修复建议

最低要求：在 main.py 中添加 HTTP Basic Auth 或 API Key 中间件。正式版本应实现基于 JWT 的用户会话，并将 operator 从 Token 中解析而非由前端传入。

CRITICAL C-03 业务人名硬编码在三处代码文件中

"请海总"、"袁广民"、"冯海林" 等人名作为字面量字符串分散在业务逻辑、数据库查询、AI 提取规则三处，导致业务数据与代码实现强耦合。人员变动时必须同步修改多处代码，且极易遗漏。

    "need_decision_by": "请海总" if "海总" in item or "决策" in item else ""

app/main.py:14–21
    PROJECT_NAME_MAP = { ..., "请海总": "冯海林" }   ← 启动时做字符串替换

特别地，main.py 中的 normalize_seed_data 函数在每次启动时将数据库中的"请海总"替换为"冯海林"——这说明 GPT 在生成代码时先用一个名字写了种子数据，后来改了人名设定，但没有统一修改，只能用启动时补丁来弥补。这是数据不一致的直接证据，也让这段逻辑在数据库已经正确后的每次重启中变为无效操作。

修复建议

2高风险问题 · HIGH
HIGH H-01 日期字段使用 String 类型，丧失时间范围查询能力

Task.plan_time、Meeting.meeting_date、Issue.expected_resolve_time 均以 String 存储日期，而非 Date 或 DateTime。这导致：无法做"本月到期"、"逾期任务"等时间范围查询；字符串比较无法保证排序正确；前端传入非法日期格式时不会被数据库拒绝。

app/models.py · 第 28, 60, 97 行
plan_time              = Column(String(20), index=True)     # Task
meeting_date           = Column(String(20), default="")    # Meeting
expected_resolve_time  = Column(String(20), default="")    # Issue
app/routers/dashboard.py · 第 27 行 — 字符串模糊匹配代替时间过滤
task_q = task_q.filter(models.Task.plan_time.like(f"{month}%"))   ← LIKE 代替 BETWEEN
修复建议

将上述字段迁移为 Column(Date) 类型，过滤查询改用 func.strftime 或 SQLAlchemy 的日期比较操作符。

HIGH H-02 结构化数据以裸 JSON 字符串存入 Text 字段

UpdateSubmission 表中有 ai_result_json、human_result_json 两个核心字段，Meeting 表中有 task_list_json、decision_items_json、risk_items_json 共 5 处用 Text 存储 JSON。这完全绕过了数据库的类型系统，导致：字段内容无法被索引；序列化错误只在运行时暴露；结构变更无迁移保护。

app/models.py · Meeting 模型
task_list_json       = Column(Text, default="")
decision_items_json  = Column(Text, default="")
risk_items_json      = Column(Text, default="")
app/routers/confirmations.py · 第 30 行 — 手动拼接 JSON 字符串
human = json.loads(row.human_result_json or row.ai_result_json or "{}")
# or "{}" 掩盖了数据为 None/空 时的结构缺失，不会报错但会返回错误数据
修复建议

SQLite 支持 JSON 亲和类型；SQLAlchemy 提供 JSON 列类型，可直接读写 Python 对象。建议将上述字段迁移为 Column(JSON)，消除手动 json.loads/dumps 调用。

HIGH H-03 Dashboard 概览接口存在 N+1 查询

/api/dashboard/overview 在遍历每个专项时，分别对 Achievement 和 Issue 表各执行一次 COUNT 查询。当专项数量为 N 时，接口共发出 2N+1 条 SQL。随数据量增长，这将成为明显的性能瓶颈。

app/routers/dashboard.py · 第 33–50 行
for project, rows in grouped.items():           ← 循环 N 次
    achievements = db.query(models.Achievement)\
        .filter(...special_project == project)\
        .count()                                    ← 第 1 条额外 SQL
    issues = db.query(models.Issue)\
        .filter(...special_project == project, ...)\
        .count()                                    ← 第 2 条额外 SQL
修复建议

使用 func.count + group_by(special_project) 一次性聚合，将 2N 条查询合并为 2 条。

HIGH H-04 全部列表接口无分页，存在全表扫描风险

/api/tasks、/api/achievements、/api/issues、/api/updates、/api/confirmations/pending 全部使用 .all() 返回全量数据。当记录数增至千条级别，每次请求将全量加载到内存并序列化，响应时间和内存占用均会急剧上升。

app/routers/tasks.py · 第 21 行（其余路由同理）
return [crud.to_dict(r) for r in q.order_by(...).all()]   ← 无 limit/offset
修复建议

在所有列表接口添加 page: int = 1、page_size: int = 20 查询参数，使用 .offset((page-1)*page_size).limit(page_size)，同时返回 total 字段供前端分页控件使用。

3中等问题 · MEDIUM
MEDIUM M-01 项目识别兜底逻辑会将无关文本错误归类

_pick_project 函数在所有关键词均未匹配时，无条件返回 "知识资产AI化"。这意味着任何不含项目关键词的文本（如纯会议记录、个人汇报）都会被错误标记为该专项，产生脏数据写入。

app/services/extractor.py · 第 16–35 行
def _pick_project(text: str) -> str:
    for project in PROJECTS:
        if project in text: return project
    aliases = { "知识": "知识资产AI化", ... }
    for word, project in aliases.items():
        if word in text: return project
    return "知识资产AI化"    ← 兜底直接写死第一个专项
修复建议

兜底返回 None 或空字符串，在上游调用处要求用户手动选择专项，而非静默写入错误值。

MEDIUM M-02 OperationLog 在关键写入路径中缺失

OperationLog 模型存在，且在 confirmations.py 中有少量使用，但 tasks/achievements/issues 的 直接增删改接口完全没有记录操作日志。审计链路存在大量盲区：通过 API 直接删除的任务不会留下任何痕迹。

app/routers/tasks.py · DELETE 路由 · 无日志
@router.delete("/{row_id}")
def delete_task(row_id: int, db: Session = Depends(get_db)):
    row = db.get(models.Task, row_id)
    db.delete(row)
    db.commit()
    # ← crud.log() 调用缺失，操作不可追溯
    return {"ok": True}
修复建议

在 crud.py 中封装 create/update/delete 通用函数，内部统一调用 crud.log，路由层使用封装函数而非直接操作 db。

MEDIUM M-03 _filtered 函数使用 hasattr 进行质量过滤，存在数据污染风险

confirmations.py 中的 _filtered 用 hasattr(model, k) 过滤前端传入的字典，保留了"模型有该属性"的所有键。由于 SQLAlchemy 模型类本身带有大量元属性，这个检查实际上并不严格，容易接受预期外的字段。此外，空字符串过滤 v != "" 会导致合法的空值无法被写入（如清空一个字段）。

app/routers/confirmations.py · 第 20–21 行
def _filtered(model, data: dict):
    return {k: v for k, v in data.items() if hasattr(model, k) and v != ""}
修复建议

使用模型的 __table__.columns.keys() 作为白名单过滤，而非 hasattr；空字符串和 None 应根据字段语义决定是否允许写入。

MEDIUM M-04 mark-unrecognized 与 reject 接口逻辑完全重复

两个接口均将 confirm_status 设为 "需修改" 并记录 reject_reason，执行路径完全相同，仅操作日志的 action 字符串不同。这是代码重复的典型表现，未来修改一处容易遗漏另一处。

app/routers/confirmations.py · 第 95–114 行
# reject
row.confirm_status = "已退回"   ← 注意：这里是"已退回"
row.reject_reason = payload.reason

# mark-unrecognized
row.confirm_status = "需修改"   ← 这里是"需修改"，两个接口实际语义不同
row.reject_reason = payload.reason   ← 但处理逻辑几乎一致

经仔细核查，两个接口的 confirm_status 值实际不同（"已退回" vs "需修改"），但代码结构完全复制粘贴，增加了维护成本和出错概率。

修复建议

提取公共辅助函数 _set_status(row, status, reason, operator)，两个接口复用，差异仅在传参。

MEDIUM M-05 种子数据含虚假文件链接，混淆演示环境真实性

seed.py 中所有成果记录的 file_link 均指向 https://example.local/bowei/{i}，这是一个无法访问的虚构域名。在演示场景下，任何点击文件链接的操作都会失败，破坏演示可信度。

app/seed.py · 第 55 行
file_link=f"https://example.local/bowei/{i + 1}",   ← 无法访问的假链接
修复建议

将链接改为空字符串，或指向本地可访问的静态资源路径，或在演示材料中说明文件系统集成为待接入状态。

总体结论

GPT 给出了一份结构上说得过去的脚手架：技术选型合理，路由分层清晰，数据模型覆盖了核心业务实体，操作日志模型也有所预留。这些是可以接受的初版起点。

然而，产品最关键的两个承诺——"AI 结构化提取"和"系统可信地记录谁做了什么"——在代码层面均未兑现。前者是空壳配合造假数值，后者因零认证导致操作人可以随意伪造。

建议处理策略：以现有代码为骨架，优先完成 C-01（接入真实 LLM）和 C-02（添加认证层）两项整改，再进行任何形式的演示或甲方汇报。C-03（去硬编码）和 H 级问题可在第一轮迭代中跟进修复。M 级问题可排入正式测试前的技术债务清单。
