# 当前系统设计说明

> 本文档是当前项目的唯一最新设计准绳。ClaudeCode、Codex 和后续开发应优先遵守本文档。
> 其他历史文档仅作背景参考，不得覆盖本文档中的业务规则。

## 1. 核心业务定位

本系统的核心不是简单记录“用户提交了什么”，而是把用户的日常更新转化为可确认、可追踪、可入库的项目推进事实。

核心链路：

```text
语音/文字/会议输入
  → AI 提取
  → 生成任务卡片
  → 提交给负责人
  → 负责人按卡片确认
  → 写入工作推进表 / 成果库 / 问题与决策
```

## 2. 三层工作推进表

工作推进表按三层组织：

```text
项目
  └─ 关键任务
      └─ 子任务
```

含义：

- 项目：承载一个专项整体目标。
- 关键任务：承载项目里的里程碑。
- 子任务：承载具体执行动作，也是成员日常提交进展的主要对象。

状态推进原则：

- 子任务有自己的状态。
- 关键任务依赖子任务进展，但子任务全部完成后不能自动关闭关键任务。
- 子任务全部完成时，应提醒负责人判断是否关闭关键任务。
- 已完成关键任务新增子任务时，关键任务应重新打开为进行中。

## 3. 提交记录、任务卡片、需处理事项

必须区分三个层级：

```text
提交记录 = 来源容器 / 收件箱外壳
任务卡片 = 负责人确认的最小业务单位
需处理事项 = 负责人判断性质的最小事项单位
```

硬规则：

- AI 确认中心不能按整条提交记录统一处理。
- 提交记录只承载多张任务卡片。
- 任务卡片才是确认、退回、转交的最小单位。
- 需处理事项只属于某张任务卡片。
- CEO 决策只针对某一条需决策事项，不针对整条提交记录。

## 4. AI 确认中心

负责人视角必须按任务卡片处理。

每张任务卡片应包含：

- 三层归属树：项目 → 关键任务 → 子任务
- 本次完成
- 下一步计划
- 成果
- 需处理事项
- 当前状态建议
- 卡片级操作区

禁止项：

- 禁止使用页面底部全局确认按钮处理整条提交。
- 禁止使用页面底部全局“请 CEO 判断”。
- 禁止在任务卡片右上角放项目下拉框作为旧式项目切换。
- 禁止把问题、风险、决策作为全局漂浮区块。

卡片操作：

```text
确认此卡入库
退回此卡修改
转交统筹人
```

如果卡片里有需处理事项：

```text
负责人逐条选择：
  问题
  风险
  待协调
  需决策
```

如果某条事项被选为需决策：

```text
负责人先判断
  → 能解决：在问题与决策页确认决策并关闭
  → 不能解决：上报 CEO
```

CEO 只看到：

- 当前任务卡片上下文
- 被标记的需决策事项
- 负责人意见

## 5. 语音更新 / 文字更新

语音更新不是单项目提交，而是：

```text
当前用户工作范围内的多任务卡片提交
```

规则：

- 一个用户可以同时负责多个项目、多个关键任务、多个子任务。
- 一次语音/文字更新可以同时描述多个项目下的多个子任务。
- AI 提取不能默认绑定当前 URL project_id。
- AI 应在当前用户可提交、负责、管理的全部工作范围内匹配。
- 每张任务卡片独立拥有自己的归属：项目 → 关键任务 → 子任务。
- 每张任务卡片都可以单独修改归属。
- 修改归属必须是三级联动：项目 → 关键任务 → 子任务。
- 候选范围必须受权限约束，不能选无权处理的项目或任务。

提交记录列表应展示任务卡片摘要，例如：

```text
文字更新 09:08 待负责人审核
包含 3 张任务卡片：
- 知识库文档检索功能设计
- 知识库首页设计
- 建议新增：知识库权限校验功能
```

提交人预览页不显示冗余的“提交人”字段。

## 6. 归属树与归属修改

负责人看到的不是项目切换器，而是任务卡片归属树。

默认展示：

```text
项目
└─ 关键任务
   └─ 子任务
```

规则：

- 归属树默认只读。
- 如果归属错误，只能点击当前卡片内的“修改归属”。
- 修改归属时在当前卡片内展开三级联动选择。
- 修改归属只影响当前任务卡片，不影响其他卡片。
- 建议新增子任务只需选择项目 → 关键任务，不需要选择已有子任务。

## 7. 问题、风险、待协调、需决策

LLM 不做最终分类。

AI 只负责发现：

```text
需处理事项
```

负责人负责定性：

```text
问题 / 风险 / 待协调 / 需决策
```

原则：

- LLM 可以描述原因、影响、建议优先级。
- LLM 不应最终决定 issue_type。
- 同一事项不能因为同时像风险和决策而重复入库。
- 入库时以负责人选择的类型为准。

## 8. 成果库

成果库不上传文件，只登记元数据和存储地址。

成果来源：

```text
1. 语音/文字更新中 AI 识别到成果
2. 用户在成果库单独登记成果
```

规则：

- 文件实体存放在外部知识库、飞书、腾讯文档或其他系统。
- 成果库只登记名称、类型、版本、所属项目、关联任务、存储地址。
- 成果可以先入库。
- 如果成果地址为空，应提醒提交人后续补录。
- 后续需要补“成果地址待补录通知”。

## 9. 子任务详情页

子任务详情应以任务进展时间线为主，而不是窄侧栏字段表。

目标结构：

```text
顶部：返回列表 / 子任务名称 / 当前状态
主区域：任务进展时间线
辅助区：所属项目、关键任务、负责人、计划时间、来源、关联成果
```

任务进展时间线每条记录展示：

- 本次完成
- 下一步计划
- 问题 / 阻塞
- 提交人
- 提交时间

## 10. 待完成事项

当前仍未完成的关键任务：

1. AI 确认中心第二阶段：卡片级数据结构补齐。
2. AI 确认中心第三阶段：后端支持单卡确认、单卡退回、单卡转交。
3. 单条需决策事项上报 CEO。
4. 语音/文字更新跨项目多任务彻底取消 URL project_id 依赖。
5. 提交记录展示任务卡片摘要。
6. 子任务详情页改成任务进展时间线主视图。
7. 成果地址待补录通知。
8. 后端原生支持 pending_items，不再依赖前端转换旧字段。
9. 问题优先级规则由负责人可调整，AI 只建议。
10. 最终全链路回归测试。

## 11. ClaudeCode 开发约束

给 ClaudeCode 的任务必须包含：

```text
请以 docs/CURRENT_SYSTEM_DESIGN.md 为准。
其他历史文档仅作背景参考，不得覆盖当前设计规则。
```

每次改动应遵守：

- 不要恢复全局确认按钮。
- 不要恢复卡片右上角项目下拉。
- 不要把提交记录当成最小操作单位。
- 不要让 LLM 最终分类问题/风险/需决策。
- 不要让语音更新绑定单一项目。
- 不要把成果库做成文件上传系统。
