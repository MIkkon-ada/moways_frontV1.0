# 语音更新新版界面与归属测试文档

本文档用于验证语音更新页新版界面是否符合当前设计：归属区小而明确，任务进展卡片作为主内容，提交后 AI确认中心看到人工修正后的项目、关键任务和子任务。

## 一、测试前准备

### 账号准备

- 普通成员账号：用于提交语音/文字更新。
- 项目负责人账号：用于在 AI确认中心审核。

### 数据准备

至少准备两个项目，并确保普通成员只参与自己有权限的项目。

示例数据：

- 项目 A：知识资产AI化
- 项目 A 关键任务：设计知识库框架、目录和管理规则
- 项目 A 子任务：知识库文档检索功能设计
- 项目 A 子任务：知识库首页设计

可选第二项目：

- 项目 B：项目统筹与复盘
- 项目 B 关键任务：确认AI升级方案、组织分工与推进机制
- 项目 B 子任务：组织AI升级启动与任务部署

## 二、界面基础测试

### VU-01 提取前页面布局

操作：

1. 普通成员登录。
2. 打开语音更新页。

预期：

- 左侧为输入区，包括录音、上传音频、粘贴文字。
- 提取模型在左侧输入区。
- 右侧为 AI 提取结果预览区。
- 页面整体不应左侧过宽、右侧过挤。

### VU-02 提取后顶部冗余提示已删除

测试文本：

```text
今天完成了知识库文档检索功能设计，检索入口已经完成，字段包括文档标题、所属项目、文档类型、关键词、上传人和更新时间。下周继续补充检索结果为空时的提示文案。状态建议保持进行中。
```

操作：

1. 粘贴上述文本。
2. 点击 AI提取。

预期：

- 不应再出现顶部蓝色 summary 提示框。
- 不应再出现“AI已提取结果，请确认无误后点击提交给负责人”的橙色提示框。
- 页面重点应落在归属面包屑和任务进展卡片上。

## 三、归属面包屑测试

### VU-03 默认归属展示

操作：

1. 使用 VU-02 的文本提取。
2. 查看“已匹配到工作推进表”区域。

预期：

- 归属区应是紧凑单行，而不是大卡片。
- 应展示类似：

```text
项 知识资产AI化 › 关 设计知识库框架、目录和管理规则 › 子 知识库文档检索功能设计
```

- 右侧应有“修改归属”按钮。
- 未展开时不应占用太多空间。

### VU-04 修改同一关键任务下的子任务

场景：

AI 识别为：

```text
知识资产AI化 › 设计知识库框架、目录和管理规则 › 知识库文档检索功能设计
```

但实际应该是：

```text
知识资产AI化 › 设计知识库框架、目录和管理规则 › 知识库首页设计
```

操作：

1. 点击“修改归属”。
2. 项目不变。
3. 关键任务不变。
4. 子任务改成“知识库首页设计”。

预期：

- 面包屑立即更新为新的子任务。
- 任务进展卡片标题也应更新或至少提交时写入新子任务。
- 提交给负责人后，AI确认中心应显示新子任务归属。

### VU-05 修改同项目下的关键任务和子任务

场景：

AI 识别为：

```text
项目A › 关键任务B › 子任务C
```

实际应该是：

```text
项目A › 关键任务E › 子任务F
```

操作：

1. 点击“修改归属”。
2. 项目保持不变。
3. 关键任务改成另一个关键任务。
4. 子任务列表应刷新为该关键任务下的子任务。
5. 选择正确子任务。

预期：

- 关键任务变化后，旧子任务选择应被清空。
- 子任务下拉只展示新关键任务下的子任务。
- 面包屑显示新的关键任务和子任务。
- 提交后 AI确认中心显示人工修正后的归属。

### VU-06 修改项目、关键任务和子任务

场景：

AI 识别为：

```text
项目A › 关键任务B › 子任务C
```

实际应该是：

```text
项目I › 关键任务K › 子任务G
```

操作：

1. 点击“修改归属”。
2. 项目改成另一个自己有权限的项目。
3. 关键任务列表应刷新。
4. 选择新项目下的关键任务。
5. 子任务列表应刷新。
6. 选择新关键任务下的子任务。

预期：

- 项目变化后，关键任务和子任务必须清空。
- 不应保留旧项目下的关键任务或子任务。
- 最终面包屑显示新的三层归属。
- 提交后 AI确认中心显示新项目、新关键任务、新子任务。

### VU-07 无权限项目不可选

操作：

1. 使用普通成员账号登录。
2. 点击“修改归属”。
3. 查看项目下拉。

预期：

- 只能看到该成员有权限参与的项目。
- 不应出现其他无权限项目。
- 如果成员只参与一个项目，项目选择可以固定或只显示一个。

## 四、任务进展卡片测试

### VU-08 单任务进展卡片主视觉

测试文本：

```text
今天完成了知识库文档检索功能设计，检索入口已经完成，结果列表也完成了初版，字段包括文档标题、所属项目、文档类型、关键词、上传人和更新时间。下周继续补充检索结果为空时的提示文案。状态建议保持进行中。
```

预期：

- 如果只有一个子任务，不需要额外显示“AI任务解析 · 1项”标题。
- 卡片标题应是子任务名称。
- 状态 badge 应显示“进行中”。
- “本次完成”应是最醒目的正文。
- “下一步计划”应和“本次完成”形成连续阅读。

### VU-09 成果字段展示

测试文本：

```text
今天完成了知识库字段命名规范V0.1，作为后续知识库页面和接口字段设计的统一参考。成果地址我后续会补录。下周继续推进字段规则落地。
```

预期：

- 任务卡片中应出现成果区域。
- 成果名称应展示清楚。
- 成果地址为空时，应显示可补录输入框或待补录提示。
- 成果地址为空不应阻塞提交给负责人。

### VU-10 问题/风险高亮

测试文本：

```text
今天推进知识库文档检索功能设计，完成了检索入口。现在有一个风险：如果本周不能确认文档类型和关键词字段的统一规则，后续检索结果可能出现分类不一致，影响联调进度。下周继续补充空状态提示。
```

预期：

- 问题/风险区域应独立展示。
- 风险应使用明显的警示样式。
- 不应和普通完成内容混在一起。

### VU-11 空字段弱化

测试文本：

```text
今天完成了知识库检索入口设计。
```

预期：

- 如果没有成果、问题、下周计划，可以显示淡灰色“未提及”。
- 空字段不应占用过多行高。
- 页面不应因为空字段显得冗余。

## 五、多任务测试

### VU-12 多子任务分卡展示

测试文本：

```text
今天完成了知识库文档检索功能设计，检索入口和结果列表已经完成，下周补充空状态提示。

另外推进了知识库首页设计，完成了首页信息架构草图，下周继续补充导航和入口布局。两个子任务状态都建议保持进行中。
```

预期：

- 应识别出多个任务更新。
- 每个子任务应独立成卡。
- 每张卡都有自己的本次完成和下一步计划。
- 多任务时可以显示“AI任务解析 · N项”。
- 不应把两个子任务的进展混成一张卡。

## 六、提交与 AI确认中心联动测试

### VU-13 提交后归属写入 AI确认中心

操作：

1. 在语音更新页提取一条内容。
2. 手动修改归属为另一个正确子任务。
3. 点击提交给负责人。
4. 切换负责人账号进入 AI确认中心。

预期：

- AI确认中心看到的是人工修正后的项目、关键任务、子任务。
- 不应回到 AI 原始识别的错误归属。
- 负责人确认入库后，工作推进表更新到正确子任务。

### VU-14 归属不完整时不能提交

操作：

1. 点击“修改归属”。
2. 只选择项目，不选择关键任务和子任务。
3. 点击提交给负责人。

预期：

- 系统应提示归属不完整。
- 不应伪造子任务归属。
- 不应提交到负责人。

### VU-15 建议新增子任务逻辑不受影响

测试文本：

```text
今天补充了知识库权限校验功能的新想法，希望后续可以按项目、文档类型和负责人控制查看范围。这个目前还没有对应到现有子任务，可以先作为建议新增子任务提交给负责人判断。
```

预期：

- 如果没有匹配到现有子任务，应显示建议新增子任务。
- 仍然需要选择归属关键任务。
- 该逻辑不应被新版三层归属选择破坏。

## 七、通过标准

本轮测试通过需要满足：

- 顶部冗余蓝色 summary 和橙色提示已删除。
- 归属区缩小为紧凑面包屑。
- 修改归属支持项目、关键任务、子任务三级级联。
- 用户只能选择自己有权限的项目。
- 任务进展卡片成为主视觉。
- 本次完成、成果、问题、下一步计划展示清晰。
- 多子任务能分卡展示。
- 提交后 AI确认中心看到人工修正后的归属。
- 归属不完整时不能提交。
