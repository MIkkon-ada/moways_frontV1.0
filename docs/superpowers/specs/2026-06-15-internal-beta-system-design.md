# 内测前系统设计说明

> 适用范围：博维 AI 升级项目驾驶舱  
> 目标：在不推翻现有实现的前提下，补齐前后端边界、统一业务语言、收敛提交-确认-入库-统计闭环，便于马上进入内测。

**Goal:** 让前端、后端和数据口径围绕同一套业务对象与状态机工作，减少“页面能看见、接口不认”“统计能出来、链路说不清”的问题。

**Architecture:** 采用“现有架构上补一层统一领域设计”的方案。前端保留 React 单页应用和现有路由结构，但将页面交互统一挂到项目上下文和稳定 API 契约上；后端保留 FastAPI + SQLAlchemy + 兼容旧字段的实现方式，但把提交、确认、入库、统计收敛到明确的业务链路和状态机。历史字段 `special_project`、`related_special_project`、旧角色字段继续兼容，但不再作为新逻辑的主语义。

**Tech Stack:** React + TypeScript + Vite, FastAPI, SQLAlchemy, SQLite/PostgreSQL, pytest, Playwright（如后续补前端 E2E）。

---

## 1. 现状判断

当前系统已经具备完整的业务链路雏形：

1. 用户提交进展、会议、语音或文本内容。
2. 后端抽取结构化建议。
3. 确认中心完成审核、打回、转交统筹、升级 CEO、CEO 批示。
4. 确认后写入任务、成果、问题等主数据。
5. Dashboard 从多张表聚合统计。

问题不在于“没有功能”，而在于“统一性不够”：

- `project_id` 与 `special_project` 并存，项目主语义不够单一。
- 确认流程是多个接口分散实现，缺少显式状态机。
- 前端权限主要用于显示控制，和后端真实权限边界的契约不够紧。
- 统计依赖多个表和历史兼容逻辑，口径容易漂移。

---

## 2. 设计原则

1. `project_id` 作为唯一主语义，旧字段仅做兼容输入和历史查询。
2. 所有提交都落到一个统一的 Submission 领域对象上。
3. 确认中心是 Submission 的状态机，不是散接口集合。
4. Task、Achievement、Issue、Meeting 是 Submission 的派生结果，不是彼此无关的表。
5. 前端负责表达和交互，后端负责真权限和业务事实。
6. 统计只认标准化后的对象与状态，不直接依赖临时拼接逻辑。

---

## 3. 总体架构

### 3.1 前端层

前端继续使用现有 React 路由体系，但需要按“项目上下文”统一组织页面：

- 登录 / 初始化
- 项目首页
- 提交入口
- 确认中心
- 任务 / 成果 / 问题 / 会议
- 设置 / 管理

前端只做三类事：

- 展示当前项目、当前角色、当前状态。
- 调用统一 API，处理加载、错误、空态。
- 按后端返回的权限字段控制按钮显隐。

前端不再自己推导业务真相，例如“这个提交现在到底能不能确认”“这个人是不是最后一个 owner”这类判断都交给后端。

### 3.2 后端层

后端建议整理成四类职责：

- API 层：接收请求、返回响应、做参数校验。
- Workflow 层：处理提交、确认、打回、转交、升级、批示、撤回。
- Policy 层：处理角色、项目边界、可见性、写权限。
- Projection 层：把 Submission 投影成 Task / Achievement / Issue / Meeting。

现有 routers 可以保留，但内部逻辑应该尽量向这四类职责收拢。

### 3.3 数据层

数据层继续使用现有表结构，但需要明确主从关系：

- `projects` 和 `people` 是基础主数据。
- `project_members` 是项目角色真相源。
- `update_submissions` 是提交与确认流程真相源。
- `tasks` / `achievements` / `issues` / `meetings` 是派生业务实体。
- `operation_logs` 记录关键行为。

历史兼容字段保留，但不再作为新功能设计的主路径。

---

## 4. 业务闭环设计

### 4.1 提交

统一提交入口仍然是 `POST /api/updates`，语义如下：

- 必须能解析出有效 `project_id`。
- `special_project` 只做兼容兜底。
- 提交后形成一条 Submission 记录，保存原始文本、抽取结果、人工修订结果、提交人、项目和状态。

建议将提交输入统一成一份标准化 DTO，避免同一个业务对象在不同页面里结构不同。

### 4.2 确认

确认中心只处理 Submission 的状态迁移，不直接承担“写哪张表”的业务决定。

建议明确状态集合：

- `pending_owner_review`
- `returned_to_submitter`
- `waiting_coordinator_feedback`
- `waiting_ceo_decision`
- `confirmed_and_stored`
- `withdrawn`
- `permanently_rejected`

不同角色只允许触发自己能做的迁移：

- owner：确认、打回、转统筹、升级 CEO
- coordinator：统筹反馈
- project_ceo：CEO 批示
- submitter：撤回、重新提交
- super_admin：绕过大部分边界

### 4.3 入库

确认通过后，Submission 的结构化结果派生出业务实体：

- `task`
- `achievement`
- `issue`
- `meeting`（如果是会议类提交）

建议保留一个“Submission -> 派生实体”的映射说明，避免统计口径和页面展示口径分离。

### 4.4 统计

Dashboard 应从标准化实体聚合，并尽量使用 Submission 的正式状态来解释提交统计。

统计口径建议固定为：

- 提交总数
- 待 owner 审核
- 已退回
- 已确认入库
- 待 CEO 决策
- 已决策
- 任务总数 / 进行中 / 已完成 / 延期
- 成果总数
- 问题总数 / 高优先级 / 待 CEO 决策

Dashboard 不要再依赖“猜字段”的方式理解数据，而应使用统一投影结果。

---

## 5. 前端设计

### 5.1 页面组织

前端页面建议保持现有结构，但按业务流收束：

- 项目上下文由 `ProjectContext` 统一提供。
- 页面只关心当前项目下的业务，不自己解析历史兼容逻辑。
- 路由层只负责把用户送到正确页面。

### 5.2 交互规则

1. 按钮显隐是 UX，不是权限。
2. 所有关键操作都要展示后端返回的错误信息。
3. 同一页面内应保持同一套状态命名，例如“待审核”“已退回”“待统筹反馈”“待 CEO 决策”。
4. 空态、错误态、加载态都要统一，不同页面不要各写一套松散文案。

### 5.3 需要统一的前端契约

建议前端和后端固定以下返回字段：

- `project`
- `currentUser`
- `roles`
- `canSubmit`
- `canConfirm`
- `canWrite`
- `visibleProjects`
- `submission`
- `humanResult`
- `aiResult`
- `workflowStatus`
- `derivedCounts`

前端后续补功能时，应尽量只消费这些稳定字段，而不是临时读深层对象。

---

## 6. 后端设计

### 6.1 API 收敛

现有路由可以保留，但建议形成三种标准模式：

- 查询类接口：只返回稳定的展示数据。
- 写入类接口：只接受标准 DTO。
- 流转类接口：只做状态迁移，不夹带额外推导。

### 6.2 推荐的内部职责划分

#### Submission Workflow

负责：

- 创建提交
- 去重
- 确认
- 打回
- 重新提交
- 撤回
- 转统筹
- CEO 决策

#### Policy Service

负责：

- 项目可见性
- 角色判断
- 写权限
- 确认中心可见性

#### Projection Service

负责：

- 确认后落任务
- 确认后落成果
- 确认后落问题
- 生成可展示的结构化结果

### 6.3 兼容策略

兼容策略应当是“输入兼容、内部统一、输出稳定”：

- 输入继续接受 `special_project`。
- 内部统一解析成 `project_id`。
- 查询旧数据时可回看兼容字段。
- 新增功能只依赖 `project_id` 和标准状态。

---

## 7. 数据模型建议

不建议马上大改表结构，但建议在设计上明确以下关系：

- `update_submissions.project_id` 是主键语义。
- `tasks / achievements / issues / meetings.project_id` 是派生实体的主归属。
- `project_members` 是权限计算的第一来源。
- `special_project` 只作为旧数据兼容标签存在。

如果后续还有一步结构优化，优先考虑增加：

- Submission 的标准化状态枚举
- Submission 到派生实体的关联追踪字段
- 统一的 workflow trace / event log

但这一步不必马上做，先把契约收紧更重要。

---

## 8. 内测验收标准

内测前至少要满足这些标准：

1. 提交入口统一，前后端字段一致。
2. 确认流程所有状态都能解释清楚。
3. owner、coordinator、project_ceo、member、super_admin 的权限边界清晰。
4. 确认后生成的任务、成果、问题能追溯到原提交。
5. Dashboard 数字与列表页数据能对得上。
6. 旧数据和兼容字段查询不崩。
7. 前端按钮显隐与后端实际权限一致，失败时提示清晰。

---

## 9. 风险点

- 历史兼容逻辑过多，容易让新逻辑被旧分支覆盖。
- 统计口径如果不先统一，内测时很容易出现“数字看起来不对但又说不清为什么”。
- 前端如果继续各页面各写权限判断，会导致按钮显隐和真实权限脱节。
- 确认流如果没有统一状态机，后续补新角色会越来越难。

---

## 10. 推荐落地顺序

1. 先统一 `project_id` 的语义。
2. 再收敛 Submission 状态机。
3. 再整理前后端契约字段。
4. 再补 Dashboard 统计口径。
5. 最后才处理更激进的数据结构优化。

这条顺序适合马上内测，因为它优先降低认知混乱，不强迫你一次性推翻现有系统。
