# 人员与登录体系设计说明

## 1. 设计目标

人员与登录体系是整个项目权限的地基。它要解决的不是单纯“能不能登录”，而是三件事：

1. 谁可以进入系统。
2. 这个人在公司和项目里是什么身份。
3. 进入某个项目后，这个人能看什么、能提交什么、能确认什么、能管理什么。

当前系统已经具备人员、登录、项目角色的雏形，但三者之间还没有完全收束成一个稳定模型。正式上线前，建议把它们拆清楚。

目标结构是：

```text
账号体系 Account
负责登录、安全、账号状态。

人员体系 Person
负责组织中的真实人员档案。

项目成员关系 ProjectMember
负责某个人在某个项目中的角色。

能力输出 Capabilities
负责告诉前端当前用户在当前项目能做什么。
```

核心原则：

```text
账号管登录
人员管身份
项目成员关系管项目角色
后端能力字段管按钮和操作
```

## 2. 当前系统现状

当前系统大致是这样：

```text
passwords.json
  username -> password_hash

auth_sessions 表
  session_id -> username

people 表
  name / role / system_role / department / contact / is_active

project_members 表
  project_id / person_id / role
```

现有优点：

- 已经有 session-cookie 登录机制。
- 已经有 `people` 人员表。
- 已经有 `project_members` 项目成员角色表。
- 前端已经能拿到当前用户、当前项目、当前项目角色。
- 一部分页面已经开始使用后端 capabilities，而不是纯前端推导权限。

当前主要问题：

- 登录账号还在 `passwords.json`，不在数据库里。
- 登录身份是 `username`，业务身份主要靠 `people.name`，二者不是强绑定。
- 改名、重名、禁用人员、禁用账号这些场景会有隐患。
- 密码哈希目前是 SHA-256，不适合作为正式上线的密码存储方案。
- 人员管理和账号管理没有明确拆分。
- 前端还有部分页面会本地推导权限，长期看容易和后端不一致。

## 3. 推荐总体模型

### 3.1 Account：登录账号

Account 只负责“能不能登录”和登录安全。

建议新增 `accounts` 表：

```text
id
username
password_hash
person_id
status
is_tech_admin
last_login_at
last_password_changed_at
failed_login_count
locked_until
created_at
updated_at
```

字段说明：

| 字段 | 含义 |
|---|---|
| `username` | 登录名，唯一 |
| `password_hash` | 密码哈希，建议 bcrypt 或 argon2 |
| `person_id` | 绑定到真实人员 |
| `status` | `active` / `disabled` / `locked` |
| `is_tech_admin` | 是否技术管理员 |
| `last_login_at` | 最近登录时间 |
| `last_password_changed_at` | 最近改密时间 |
| `failed_login_count` | 连续失败次数 |
| `locked_until` | 锁定截止时间 |

建议不要把账号状态和人员状态混在一起。

例如：

```text
人员还在职，但账号被禁用
=> Person.is_active = true
=> Account.status = disabled

人员离职
=> Person.is_active = false
=> Account.status = disabled
```

### 3.2 Person：人员档案

Person 负责“这个人是谁”。

当前 `people` 表已经有基础字段，可以继续保留：

```text
id
name
department
role / position
contact
system_role
is_active
```

建议后续优化：

```text
employee_no
display_name
email
phone
avatar
joined_at
left_at
```

这里要注意：

```text
Person.name 是展示名，不应该作为权限判断的长期主键。
```

权限和关联应该尽量走 `person_id`，而不是走姓名字符串。

### 3.3 ProjectMember：项目角色

ProjectMember 负责“某个人在某个项目里是什么角色”。

当前系统已经有这个表，方向是对的：

```text
project_id
person_id
role
```

建议项目角色保持这几类：

| 角色 | 含义 |
|---|---|
| `owner` | 项目负责人 |
| `coordinator` | 项目统筹人 |
| `member` | 项目成员 / 协同人 |
| `project_ceo` | 项目 CEO / 决策人 |

不要把“公司职务”直接当成项目权限。

例如：

```text
张三在公司是顾问
但在 ProjectA 是 member
在 ProjectB 是 owner
```

所以公司身份和项目角色必须分开。

## 4. 登录链路设计

推荐正式链路：

```text
用户输入 username/password
        ↓
查询 accounts
        ↓
校验账号状态
        ↓
校验密码 hash
        ↓
找到绑定的 person
        ↓
检查 Person.is_active
        ↓
创建 auth_session
        ↓
返回 cookie
        ↓
前端调用 /api/people/me 获取身份上下文
```

`/api/people/me` 建议返回：

```json
{
  "account_id": 1,
  "person_id": 3,
  "name": "moways",
  "system_role": "普通成员",
  "is_tech_admin": false,
  "is_ceo": false,
  "is_process_guard": false,
  "visible_projects": ["ProjectA"],
  "project_roles": {
    "ProjectA": "owner"
  }
}
```

更理想的是同时返回当前用户所有项目角色：

```json
{
  "projects": [
    {
      "project_id": 1,
      "project_name": "ProjectA",
      "roles": ["owner"]
    },
    {
      "project_id": 2,
      "project_name": "ProjectB",
      "roles": ["member"]
    }
  ]
}
```

## 5. 权限设计

权限不要只靠前端判断。正式设计应当分三层。

### 5.1 登录权限

判断是否能进入系统：

```text
Account.status == active
Person.is_active == true
session 未过期
```

### 5.2 全局权限

全局权限控制平台级能力。

例如：

| 权限 | 来源 |
|---|---|
| 技术管理员 | `Account.is_tech_admin` |
| 组长 CEO | `Person.system_role` |
| 过程保障 | `Person.system_role` |

建议：

```text
技术管理员放 Account 上
业务身份放 Person.system_role 上
项目内权限放 ProjectMember 上
```

### 5.3 项目权限

项目权限从 `project_members` 计算：

```text
owner
coordinator
member
project_ceo
```

不要再长期依赖 `special_project_duty`、`project_roles` 字符串字段做核心判断。它们可以作为兼容层，但不应继续扩展业务。

### 5.4 能力字段 Capabilities

前端最终不应该自己猜“这个按钮能不能点”。

后端应该输出当前项目能力：

```json
{
  "roles": ["owner"],
  "canSubmit": true,
  "canConfirm": true,
  "canCoordinate": false,
  "canManageTask": true,
  "canManageTrash": true,
  "canManageProjectMembers": false,
  "canViewCenter": true
}
```

这样前端只负责消费能力字段。

## 6. 人员管理页面应该怎么设计

人员管理建议分成两个视角。

### 6.1 人员档案

用于维护公司人员。

字段：

```text
姓名
部门
岗位 / 职务
联系方式
系统身份
是否在职
备注
```

操作：

```text
新增人员
编辑人员
停用人员
查看参与项目
```

### 6.2 账号管理

用于维护登录账号。

字段：

```text
登录名
绑定人员
账号状态
是否技术管理员
最近登录时间
密码更新时间
失败登录次数
```

操作：

```text
创建账号
绑定人员
重置密码
禁用账号
解锁账号
强制下线
```

注意：

```text
不是每个 Person 都必须有 Account。
```

例如外部协作人可以存在于人员表中，但不一定允许登录系统。

## 7. 项目成员管理应该怎么设计

项目成员管理应该挂在项目下面，而不是挂在账号下面。

在项目详情里配置：

```text
项目负责人 owner
项目统筹 coordinator
项目成员 member
项目 CEO project_ceo
```

一个人可以在同一个项目拥有多个角色，但要谨慎。

建议规则：

```text
一个项目至少 1 个 owner
coordinator 可以为空
member 可以多个
project_ceo 可以为空或 1 个
技术管理员默认可管理全部项目，但不必写入 project_members
```

## 8. 安全设计建议

正式上线建议补这些能力：

### 8.1 密码哈希

当前 SHA-256 不适合正式密码存储。

建议改为：

```text
bcrypt
或 argon2
```

密码字段不要明文，不要可逆加密。

### 8.2 登录失败锁定

建议规则：

```text
连续失败 5 次，锁定 15 分钟
管理员可手动解锁
成功登录后清零失败次数
```

### 8.3 Session 管理

当前已有 `auth_sessions` 表，可以继续扩展：

```text
ip_address
user_agent
revoked_at
revoked_reason
```

可支持：

```text
查看在线会话
强制下线
修改密码后清除旧 session
```

### 8.4 首次登录改密

新增字段：

```text
must_change_password
```

管理员创建账号后给临时密码，用户首次登录必须修改。

## 9. 和现有系统的迁移关系

不建议一次推倒重做。

推荐兼容迁移：

### 阶段 1：新增 accounts 表

保留 `passwords.json`，新增数据库账号表。

登录时：

```text
优先查 accounts
查不到再回退 passwords.json
```

这样不会影响当前内测账号。

### 阶段 2：账号绑定人员

给每个账号补 `person_id`。

如果现有 `username` 和 `people.name` 一致，可以自动绑定。

不一致的账号由管理员手动绑定。

### 阶段 3：/api/people/me 改为基于 account + person

登录后身份上下文从：

```text
session.username
```

升级为：

```text
session.account_id -> account.person_id -> person -> project_members
```

### 阶段 4：收敛权限判断

逐步把后端散落的角色判断改成统一服务：

```text
auth service
person service
policy service
capabilities service
```

### 阶段 5：移除 passwords.json

当所有账号都迁移到数据库后，再移除 `passwords.json` 登录回退。

## 10. 推荐落地顺序

### P0：必须先做

```text
1. 新增 accounts 表
2. 登录优先走 accounts
3. accounts 绑定 person_id
4. /api/people/me 返回 account_id/person_id
5. 管理员可禁用账号
```

### P1：正式上线前建议做

```text
1. bcrypt / argon2 替换 SHA-256
2. 重置密码
3. 修改密码
4. 首次登录强制改密
5. 登录失败锁定
6. 修改密码后清除旧 session
```

### P2：体验完善

```text
1. 账号管理页面
2. 人员详情页显示参与项目
3. 项目成员管理更清晰
4. 在线 session 管理
5. 操作日志记录账号管理动作
```

## 11. 推荐最终业务规则

### 登录

```text
只有 active account 可以登录
绑定的 person 必须 active
密码错误次数过多则临时锁定
```

### 人员

```text
人员可以没有账号
账号必须绑定人员
人员停用后，其账号默认不可登录
```

### 项目角色

```text
项目角色只从 project_members 判断
不要依赖姓名字符串长期判断权限
```

### 技术管理员

```text
技术管理员是平台维护身份
不必加入每个项目
默认可看、可管全局配置
但业务操作最好仍记录实际操作人
```

### 前端权限

```text
前端只根据后端 capabilities 控制按钮显示
后端接口必须再次校验权限
```

## 12. 一句话总结

当前系统已经有基础，但正式上线前需要从：

```text
username + passwords.json + 人员姓名匹配
```

升级为：

```text
Account 登录账号
+ Person 人员档案
+ ProjectMember 项目角色
+ Capabilities 能力输出
```

这样人员、登录、权限、项目角色才能长期稳定，不会因为改名、重名、离职、禁用账号、跨项目多角色而变乱。
