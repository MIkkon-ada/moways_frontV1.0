# 语音识别准确率优化设计说明

## 1. 设计目标

语音更新模块的目标不是单纯“把声音转成文字”，而是让项目成员用自然语言提交工作进展，并且能够稳定进入后续的 AI 提取、负责人确认、工作推进表入库流程。

所以语音识别链路要解决三件事：

1. 尽可能准确地识别用户原话，尤其是项目名、人名、关键任务名、成果名、问题描述。
2. 识别失败或低置信度时要明确提示用户，不要让错误文本继续进入 AI 提取。
3. 识别结果要能和工作推进表挂钩，但不能越权猜项目、不能凭空创造项目。

核心原则：

```text
录音质量先保证
ASR 只负责忠实转写
LLM 只负责轻量纠错和结构化提取
项目归属由用户权限和明确选择决定
低质量结果必须拦截或提示确认
```

## 2. 当前系统现状

当前语音链路大致是：

```text
浏览器录音
  -> MediaRecorder 生成 webm/opus
  -> 前端转成 16k WAV
  -> POST /api/transcribe
  -> 后端调用 DashScope paraformer-realtime-v2
  -> 返回 text
  -> 前端把 text 送入 AI 提取
  -> 用户确认后提交给负责人
```

当前相关位置：

```text
frontend/src/pages/VoiceUpdatePage.tsx
  录音、上传、转写、AI 提取入口

bowei_ai_dashboard/app/routers/transcribe.py
  /api/transcribe 后端转写接口

bowei_ai_dashboard/app/services/extractor.py
  AI 提取结构化字段
```

现有优点：

- 已经有独立的 `/api/transcribe` 接口。
- 已经支持录音和上传音频两种入口。
- 已经接入 DashScope ASR。
- 语音识别和 AI 提取是分开的，后续可以分层优化。
- 识别后的文本仍需要用户确认，具备人工兜底空间。

当前主要问题：

- 前端会把浏览器录音二次转成 16k WAV，可能损失音质和细节。
- 后端使用的是 `paraformer-realtime-v2`，更偏实时识别，不一定适合所有录音文件场景。
- 麦克风采集没有明确配置降噪、回声消除、自动增益等约束。
- 没有业务热词，项目名、人名、任务名、成果名容易识别错。
- ASR 结果没有做业务词纠错，错字会直接进入 AI 提取。
- 没有识别质量分层，例如“高置信可继续、低置信需人工确认、失败需重录”。
- 语音更新和工作推进表的绑定逻辑还需要更明确：语音输入可以提取内容，但不能擅自决定用户没有权限的项目归属。

## 3. 为什么准确率低

### 3.1 录音源质量不稳定

浏览器默认 `getUserMedia({ audio: true })` 会交给浏览器自行选择采集参数，不同电脑、浏览器、麦克风的结果可能差异很大。

常见问题：

- 环境噪声大。
- 麦克风离说话人远。
- 浏览器自动增益把音量拉爆。
- 会议室多人说话导致串音。
- 用户边思考边说，停顿、重复、口头禅较多。

### 3.2 前端二次转码可能损失信息

当前前端录音后会把 `webm/opus` 解码，再用 `OfflineAudioContext` 重采样为 16k WAV。

这个过程有风险：

- 浏览器解码和重采样质量不可控。
- 双声道转单声道可能丢失部分信息。
- 原始 opus 已经压缩，再转 WAV 不会提升音质。
- 文件扩展名变成 WAV 后，后端固定按 16k 处理，容易掩盖真实采样率问题。

建议长期改为：

```text
前端保留原始录音文件
后端统一接收原始音频
后端负责格式判断、必要转码、ASR 调用
```

### 3.3 ASR 模型场景不完全匹配

当前后端使用：

```text
model = paraformer-realtime-v2
```

这个模型适合实时或准实时场景，但语音更新和会议音频更像“录音文件转写”。如果正式上线追求准确率，应该评估更适合文件转写、长音频、标点和热词的模型或接口。

### 3.4 缺少业务热词

这个系统里有大量普通 ASR 不熟悉的词：

```text
项目名：PROjectA、项目B
人员名：moways、刘万超、张三
关键任务名：自动化派生任务、持续推进专项工作
成果名：自动化派生成果
业务词：确认中心、退回修改、工作推进表、成果库、统筹人
```

没有热词时，ASR 可能把这些词识别成读音相近但业务上完全错误的词。

### 3.5 ASR 错误会放大到 AI 提取

AI 提取依赖转写文本。如果 ASR 已经把项目名、任务名、人名识别错，后面的 LLM 很可能会基于错误文本继续提取。

也就是说，问题不是“AI 提取提示词写得不够聪明”，而是：

```text
声音 -> 文本 这一层错了
结构化提取只能在错误文本上继续推理
```

所以必须在 ASR 和结构化提取之间增加一层“业务词纠错/质量检查”。

## 4. 推荐目标链路

正式上线建议把语音更新链路改成六段：

```text
1. 音频采集
   浏览器尽量采集干净、稳定的原始音频

2. 音频上传
   前端上传原始录音或原始文件，不在前端强制转 WAV

3. ASR 转写
   后端选择适合录音文件的 ASR 模型或接口

4. 业务词纠错
   用项目、人名、任务名、成果名做热词和轻量 LLM 校正

5. 结构化提取
   从纠错后的文本中提取任务、完成事项、成果、问题、下一步计划

6. 人工确认
   用户确认项目归属和提取结果后，再提交给负责人
```

目标结构：

```text
VoiceUpdatePage
  -> upload original audio

/api/transcribe
  -> validate file
  -> call ASR
  -> return raw_text + quality metadata

/api/transcribe/correct 或内部服务
  -> use business glossary
  -> return corrected_text + changed_terms

/api/updates/extract
  -> LLM structure extraction
  -> never guess unauthorized project

/api/updates
  -> create submission
  -> wait for confirmation
```

## 5. 项目归属规则

语音识别优化不能绕开工作推进表的核心规则。

用户提交语音更新时，项目归属应该按下面顺序判断：

```text
第一优先级：用户在页面明确选择的项目
第二优先级：用户输入中明确提到的项目，且用户对该项目有提交权限
第三优先级：无法确定时留空或要求用户选择
```

不能做的事：

```text
不能因为只有一个默认项目就自动塞进去
不能靠 special_project 文本决定 project_id
不能把用户没有权限的项目写入提交
不能让 LLM 猜测“最像哪个项目”
```

推荐规则：

| 场景 | 系统行为 |
|---|---|
| 用户已选择项目 | 以用户选择的 `project_id` 为准 |
| 用户未选择项目，但文本明确提到项目 | 后端验证用户是否有该项目提交权限 |
| 用户未选择项目，文本也没提项目 | 提示用户选择项目 |
| 文本提到多个项目 | 要求用户确认归属，不能自动提交 |
| 文本提到无权限项目 | 阻止提交，并提示无权限 |

## 6. 业务热词设计

热词不应该写死在代码里，应该从系统数据中动态生成。

建议热词来源：

```text
projects.name
people.name
tasks.title
achievements.title
issues.title
project_members 关联的人名
常用业务词表
```

热词按当前用户权限过滤：

```text
用户能访问哪些项目
  -> 只取这些项目下的人名、任务名、成果名、问题名
  -> 生成本次 ASR 和纠错可用的业务词表
```

这样可以避免两个问题：

- 识别时把用户无权限项目的词也引入上下文。
- 多公司、多项目部署时业务词互相污染。

## 7. LLM 纠错层设计

ASR 后可以增加一个轻量 LLM 纠错步骤，但它不能重新写内容。

纠错层职责：

```text
修正明显错字
修正业务专有名词
补全标点
保持原意
不新增事实
不提取结构化字段
不猜测项目归属
```

建议 Prompt 原则：

```text
你是语音转写文本校对器。
只能根据给定业务词表修正明显识别错误。
不得新增原文没有的信息。
不得改变任务状态、责任人、项目归属。
如果不确定，保留原文。
返回 corrected_text 和 changed_terms。
```

返回结构建议：

```json
{
  "raw_text": "原始 ASR 文本",
  "corrected_text": "纠错后的文本",
  "changed_terms": [
    {
      "from": "识别错的词",
      "to": "业务词",
      "reason": "命中当前项目热词"
    }
  ],
  "needs_user_review": false
}
```

## 8. 质量判断规则

建议给语音识别结果增加质量状态，而不是只返回一段文本。

```text
good
  可以继续进入 AI 提取

review
  可以展示，但需要用户重点确认

failed
  不进入 AI 提取，提示用户重录或手动输入
```

判断维度：

| 维度 | 说明 |
|---|---|
| 文本长度 | 太短可能是未识别到有效语音 |
| 业务词命中率 | 项目、任务、人名完全没命中时要提示 |
| 重复片段 | 大量重复说明识别质量差 |
| 无意义字符 | 符号、乱码、断句异常 |
| ASR 错误 | 模型返回错误或空文本 |
| 用户确认 | 用户可以手动选择“文本正确”继续 |

前端展示建议：

```text
识别成功：展示转写文本和“继续 AI 提取”
需要确认：提示“识别结果可能不准确，请先检查”
识别失败：提示“未识别到有效语音，请重录或手动输入”
```

## 9. 分阶段改造建议

### 第一阶段：低风险快速修正

目标：不大改后端模型，先减少前端造成的音质损失。

改动：

- 浏览器录音增加音频约束：

```ts
{
  audio: {
    echoCancellation: true,
    noiseSuppression: true,
    autoGainControl: true,
    channelCount: 1
  }
}
```

- 前端录音完成后直接上传原始 `webm/opus`，不再强制转成 WAV。
- 后端继续根据文件扩展名判断格式。
- 前端展示“转写文本确认区”，让用户在 AI 提取前可以编辑。

收益：

- 改动小。
- 风险低。
- 立刻减少二次转码损耗。

### 第二阶段：业务词纠错

目标：提升项目名、人名、任务名、成果名的识别准确性。

改动：

- 后端新增业务词表生成服务。
- `/api/transcribe` 返回 `raw_text` 和 `corrected_text`。
- 增加 LLM 轻量纠错层。
- 前端展示纠错差异，让用户能看到系统改了哪些词。

收益：

- 对当前业务最有价值。
- 能明显减少“项目名/任务名识别错导致提取错”的问题。

### 第三阶段：替换或增强 ASR 模型

目标：为正式上线选择更适合录音文件、长音频和会议音频的 ASR 方案。

改动：

- 调研并接入更适合文件转写的 ASR 模型或接口。
- 区分短语音更新和会议音频：

```text
短语音更新：快速识别，低延迟
会议音频：文件转写，支持长音频、说话人、时间戳
```

- 后端保留统一 `/api/transcribe` 门面，前端不感知具体模型变化。

收益：

- 长期准确率更稳定。
- 后续会议纪要、任务提取也能复用。

## 10. 和工作推进表的关系

语音更新是输入层，不是工作推进表本身。

正确关系：

```text
语音/会议输入
  -> ASR 转写
  -> LLM 纠错
  -> AI 提取
  -> 用户确认
  -> 提交收件箱 UpdateSubmission
  -> 负责人确认
  -> 工作推进表 KeyTask/SubTask/Achievement/Issue
```

语音输入可以产生：

```text
子任务进展
成果说明
问题上报
下一步计划
协同请求
决策请求
```

但语音输入不能直接绕过确认流修改工作推进表。

尤其是：

```text
不能直接关闭关键任务
不能直接改变项目归属
不能直接把问题标记为解决
不能直接创建无权限项目的数据
```

## 11. 验收用例

| 编号 | 用例 | 期望结果 |
|---|---|---|
| V01 | 用户录一段清晰语音 | 能识别出主要内容 |
| V02 | 用户说出项目名和关键任务名 | 项目名、任务名不被错误改写 |
| V03 | 用户没有说项目名，但页面已选择项目 | 提交归属为页面选择项目 |
| V04 | 用户说出另一个无权限项目 | 系统阻止提交或要求重新选择 |
| V05 | 用户说多个项目 | 系统提示确认归属，不自动提交 |
| V06 | 录音只有噪声或空白 | 不进入 AI 提取，提示重录 |
| V07 | ASR 识别出明显错字 | 纠错层按业务词表修正，并展示修正记录 |
| V08 | 用户手动修改转写文本后再提取 | AI 使用用户修改后的文本 |
| V09 | 会议音频上传 | 能走同一转写接口，但可使用更适合长音频的处理策略 |
| V10 | 热词中存在多个相近任务名 | 不强行猜测，低置信时要求用户确认 |

## 12. 给 ClaudeCode/Codex 的实施指令

可以按下面顺序执行，不建议一次性大改：

```text
第一步：
只改 VoiceUpdatePage.tsx 的录音采集和上传逻辑。
增加麦克风约束。
停止前端 encodeToWav 转码。
录音结束后直接上传原始 webm/opus 文件。
保留上传文件逻辑不变。
运行前端 build 验证。

第二步：
改造 /api/transcribe 返回结构。
从 { text } 扩展为 { raw_text, corrected_text, text, quality, changed_terms }。
为了兼容前端，text 仍返回最终可展示文本。

第三步：
新增业务词表服务。
根据当前用户可见项目生成项目名、人名、关键任务名、成果名、问题名热词。
注意必须按用户权限过滤。

第四步：
新增 ASR 后的 LLM 纠错层。
纠错只允许修正明显识别错误，不允许新增事实和猜项目。

第五步：
为语音更新页增加质量提示。
good 直接允许 AI 提取。
review 要求用户确认或编辑。
failed 不允许继续 AI 提取。
```

必须注意：

```text
不要改确认中心状态机。
不要改工作推进表入库逻辑。
不要让 special_project 决定 project_id。
不要删除现有手动输入能力。
不要把 ASR 纠错和 AI 提取混成一个函数。
```

## 13. 总结判断

目前语音识别准确率低，主要不是单一提示词问题，而是语音链路本身缺少正式上线所需的音频质量控制、业务热词、纠错层和质量拦截。

最值得先做的是：

```text
停止前端强制转 WAV
上传原始音频
增加麦克风采集约束
增加业务词纠错
```

这几项不会推翻现有系统，也不会破坏工作推进表主链路，但能显著改善“识别错 -> 提取错 -> 提交错”的连锁问题。
