# 工作推进表状态与权限流转设计

## 1. 设计目标

工作推进表是系统的核心业务事实表，不是普通待办清单，也不是用户原始提交记录。

本设计的目标是把项目推进过程拆清楚：

- 项目承载整体目标。
- 关键任务承载项目里程碑。
- 子任务承载具体执行动作。
- 语音更新、文字更新、会议纪要属于输入层。
- AI确认中心负责把输入层内容转成待确认事实。
- 工作推进表只保存负责人确认后的推进事实。

最终要解决的问题是：

- AI不能直接改最终业务事实。
- 子任务负责人不能直接关闭关键任务。
- 子任务全部完成后，关键任务不能自动关闭。
- 项目、关键任务、子任务三层状态要有统一规则。
- 权限判断不能继续依赖模糊的“成员”概念。

## 2. 核心对象

### 2.1 项目

项目是最高层业务对象，代表一个公司级或部门级专项。

项目字段建议：

| 字段 | 含义 |
|---|---|
| project_id | 项目唯一ID |
| name | 项目名称 |
| owner | 项目负责人 |
| coordinator | 项目统筹人 |
| ceo | 项目CEO或决策人 |
| status | 项目状态 |
| progress | 项目进度 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

项目状态不建议由普通用户直接修改，应由关键任务汇总后提示项目负责人确认。

### 2.2 关键任务

关键任务是项目里的里程碑，代表项目推进中的阶段性目标。

关键任务字段建议：

| 字段 | 含义 |
|---|---|
| task_id | 关键任务ID |
| project_id | 所属项目 |
| title | 关键任务名称 |
| owner | 项目负责人，默认继承项目负责人 |
| coordinator | 项目统筹人，默认继承项目统筹人 |
| status | 关键任务状态 |
| completion_criteria | 完成标准 |
| planned_start | 计划开始时间 |
| planned_end | 计划结束时间 |
| result_count | 关联成果数 |
| issue_count | 关联问题数 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

关键任务不等于子任务集合的简单求和。子任务全部完成，只能说明“可以提醒负责人确认关闭”，不能自动说明关键任务已经完成。

### 2.3 子任务

子任务是实际执行单元，必须有明确负责人。

子任务字段建议：

| 字段 | 含义 |
|---|---|
| subtask_id | 子任务ID |
| task_id | 所属关键任务 |
| project_id | 所属项目，冗余保存便于查询 |
| title | 子任务名称 |
| owner | 子任务负责人 |
| status | 子任务状态 |
| planned_start | 计划开始时间 |
| planned_end | 计划结束时间 |
| latest_update | 最新进展摘要 |
| created_from | 来源：手动创建 / AI确认写入 / 大纲导入 |
| source_submission_id | 来源提交ID |
| created_at | 创建时间 |
| updated_at | 更新时间 |

子任务下发给负责人后，默认应进入“进行中”。如果子任务没有负责人，则保持“未开始”。

## 3. 角色模型

系统不再把“成员”作为权限角色。

“成员”只表示某个人参与了项目，但权限判断要看他在具体对象上的身份。

正式角色建议如下：

| 角色 | 定义 |
|---|---|
| 项目负责人 | 项目和关键任务的最终确认人 |
| 项目统筹人 | 协调推进、处理阻塞、协助负责人 |
| 子任务负责人 | 具体执行人，负责提交进展、完成、问题 |
| 项目CEO | 查看进展、处理重大决策，不承担日常执行 |
| 技术管理员 | 系统级管理和兜底权限 |

判断一个用户能不能操作，不应问“他是不是成员”，而应问：

- 他是不是这个项目的负责人？
- 他是不是这个项目的统筹人？
- 他是不是这个子任务的负责人？
- 他是不是这个项目的CEO？
- 他是不是技术管理员？

## 4. 状态模型

项目、关键任务、子任务统一使用五个主状态：

| 状态 | 含义 |
|---|---|
| 未开始 | 尚未进入实际推进 |
| 进行中 | 已经开始推进，或有子级任务正在推进 |
| 已完成 | 已由负责人确认关闭 |
| 延期 | 计划时间已受影响，或存在延期子级任务 |
| 暂缓 | 因资源、决策、外部条件等原因暂停 |

可以保留“已归档”作为系统管理状态，但它不属于日常推进五态。

## 5. 子任务状态切换规则

子任务是最小执行单元，状态建议由AI给出，最终由项目负责人确认后入库。

| 当前状态 | 目标状态 | 触发条件 | 发起人 | 是否直接生效 |
|---|---|---|---|---|
| 未开始 | 进行中 | 子任务被分配给负责人 | 项目负责人/统筹人/管理员 | 可以直接生效 |
| 进行中 | 进行中 | 提交了进展，仍有后续计划 | 子任务负责人 | 否，进AI确认中心 |
| 进行中 | 已完成 | 明确提交完成，且无后续遗留 | 子任务负责人 | 否，进AI确认中心 |
| 进行中 | 延期 | 明确延期，或计划时间已超 | 子任务负责人/负责人/统筹人 | 需负责人确认 |
| 进行中 | 暂缓 | 等待资源、决策、外部条件 | 子任务负责人/负责人/统筹人 | 需负责人确认 |
| 暂缓 | 进行中 | 阻塞解除，恢复推进 | 子任务负责人/负责人/统筹人 | 需负责人确认 |
| 延期 | 进行中 | 延期后恢复推进 | 子任务负责人/负责人/统筹人 | 需负责人确认 |
| 延期 | 已完成 | 延期后完成 | 子任务负责人 | 否，进AI确认中心 |
| 已完成 | 进行中 | 重新打开继续推进 | 项目负责人/管理员 | 需要填写原因 |

### AI对子任务状态的推荐规则

AI只做状态建议，不直接写最终状态。

| 输入内容特征 | AI建议状态 |
|---|---|
| 有“下一步”“后续继续”“还需要”等表达 | 进行中 |
| 有“已完成”“全部完成”“无遗留”等表达 | 已完成 |
| 有“等确认”“等资源”“暂停”“卡住”等表达 | 暂缓 |
| 有“延期”“赶不上”“超过计划”等表达 | 延期 |
| 内容只是补充信息，无法判断状态 | 保持当前状态 |

AI建议必须允许提交人在草稿里修改。

## 6. 关键任务状态切换规则

关键任务是里程碑，不能被子任务自动关闭。

| 条件 | 关键任务状态 |
|---|---|
| 没有子任务，或所有子任务未开始 | 未开始 |
| 任一子任务进行中或已完成 | 进行中 |
| 任一子任务延期 | 延期 |
| 任一子任务暂缓，且没有延期子任务 | 暂缓 |
| 所有子任务已完成 | 保持进行中，并提醒负责人确认是否关闭 |
| 项目负责人确认关闭 | 已完成 |
| 已完成后新增子任务 | 自动重新打开为进行中 |

关键任务关闭条件：

- 必须至少存在一个子任务，或负责人主动说明无需子任务。
- 所有关联子任务必须为已完成。
- 项目负责人或技术管理员确认后，关键任务才能变为已完成。

如果关键任务已完成后又新增子任务，说明该关键任务仍有实际推进内容，应自动重新打开为“进行中”，并写入操作日志。

## 7. 项目状态切换规则

项目状态由关键任务汇总，但项目完成也不建议自动关闭。

| 条件 | 项目状态 |
|---|---|
| 所有关键任务未开始 | 未开始 |
| 任一关键任务进行中或已完成 | 进行中 |
| 任一关键任务延期 | 延期 |
| 任一关键任务暂缓，且没有延期关键任务 | 暂缓 |
| 所有关键任务已完成 | 提醒项目负责人确认项目完成 |
| 项目负责人确认项目完成 | 已完成 |

项目完成应由负责人确认，因为项目完成通常还涉及验收、成果归档、复盘和对外交付。

## 8. AI确认中心流转规则

AI确认中心是输入层和工作推进表之间的闸口。

标准流程：

```text
用户提交语音/文字/会议内容
  ↓
AI提取事实
  ↓
匹配项目、关键任务、子任务
  ↓
生成状态建议、成果、问题、下步计划
  ↓
提交人修改草稿
  ↓
提交给项目负责人
  ↓
项目负责人确认
  ↓
写入工作推进表、成果库、问题库和操作日志
```

AI确认中心必须区分以下结果类型：

| 类型 | 处理方式 |
|---|---|
| 匹配已有子任务的进展 | 更新该子任务最新进展和状态建议 |
| 匹配已有子任务的完成 | 提交负责人确认是否完成 |
| 未匹配子任务但像具体执行动作 | 标记为“建议新增子任务” |
| 关键任务级问题 | 挂到关键任务问题区 |
| 项目级问题 | 挂到项目问题区 |
| 成果 | 写入成果库或关键任务成果 |
| 无法判断 | 标记为“待负责人判断” |

AI不能因为用户提到一个新事项，就直接把它塞进项目下成为正式任务。必须先经过负责人确认。

## 9. 权限矩阵

| 操作 | 子任务负责人 | 项目负责人 | 项目统筹人 | 项目CEO | 技术管理员 |
|---|---|---|---|---|---|
| 查看项目 | 相关项目可看 | 可看 | 可看 | 可看 | 可看全部 |
| 查看工作推进表 | 相关项目可看 | 可看 | 可看 | 可看 | 可看全部 |
| 提交子任务进展 | 可提交自己的 | 可提交 | 可提交 | 不建议 | 可提交 |
| 修改AI草稿 | 可修改自己的提交 | 可修改相关提交 | 可协助修改 | 不建议 | 可修改 |
| 确认子任务状态 | 不可直接确认 | 可确认 | 可协助，不最终确认 | 不可 | 可确认 |
| 新增子任务 | 提交建议 | 可直接新增 | 可建议或协助 | 不建议 | 可直接新增 |
| 修改子任务 | 提交建议 | 可修改 | 可协助 | 不建议 | 可修改 |
| 删除子任务 | 不可 | 可软删除 | 不建议 | 不可 | 可软删除 |
| 恢复子任务 | 不可 | 可恢复 | 不建议 | 不可 | 可恢复 |
| 关闭关键任务 | 不可 | 可关闭 | 可提醒 | 不可 | 可关闭 |
| 删除关键任务 | 不可 | 可软删除 | 不建议 | 不可 | 可软删除 |
| 恢复关键任务 | 不可 | 可恢复 | 不建议 | 不可 | 可恢复 |
| 查看回收站 | 不可 | 可看本项目 | 不可 | 不可 | 可看全部 |
| CEO决策批示 | 不可 | 可发起 | 可发起 | 可批示 | 可管理 |

删除、批量删除、回收站恢复、回收站查看，建议只给项目负责人和技术管理员。

## 10. 前端交互规则

### 10.1 工作推进表

工作推进表应按三层展示：

```text
项目
  └─ 关键任务
       └─ 子任务
```

关键任务行展示：

- 关键任务名称
- 项目负责人
- 计划时间
- 当前状态
- 子任务完成数
- 风险状态
- 查看详情

子任务行展示：

- 子任务名称
- 子任务负责人
- 计划时间
- 当前状态
- 删除或恢复入口

不建议在列表上铺太多操作按钮。删除、恢复、重新打开等高风险操作应放到详情抽屉或更多菜单里，并且必须二次确认。

### 10.2 子任务全部完成提醒

当某个关键任务下所有子任务完成时：

- 不自动关闭关键任务。
- 在关键任务详情里提示“所有子任务已完成，请负责人确认是否关闭关键任务”。
- 项目负责人看到“确认关闭关键任务”按钮。
- 非负责人只看到提示，不看到关闭按钮。

### 10.3 新增子任务导致重新打开

当关键任务已完成后新增子任务：

- 前端提示“该关键任务已完成，新增子任务后将重新打开为进行中”。
- 负责人确认后新增。
- 后端把关键任务状态改为“进行中”。
- 写入操作日志。

## 11. 后端校验规则

后端必须是权限和状态的最终裁判。

前端按钮显隐只能改善体验，不能作为安全边界。

后端需要统一校验：

- 当前用户是否是项目负责人。
- 当前用户是否是技术管理员。
- 当前用户是否是子任务负责人。
- 当前操作是否允许从当前状态切到目标状态。
- 当前对象是否已软删除。
- 当前对象是否属于用户有权限的项目。

建议抽成三个服务：

| 服务 | 职责 |
|---|---|
| policy service | 判断谁能做什么 |
| workflow service | 判断状态能不能切换 |
| projection service | 汇总项目、关键任务、子任务状态 |

## 12. 操作日志规则

以下操作必须写入操作日志：

- AI确认写入
- 子任务状态变化
- 关键任务状态变化
- 项目状态变化
- 新增子任务
- 删除子任务
- 恢复子任务
- 删除关键任务
- 恢复关键任务
- 关键任务重新打开
- 负责人确认关闭关键任务

操作日志建议只增不改。

字段建议：

| 字段 | 含义 |
|---|---|
| entity_type | project / task / subtask / issue / achievement |
| entity_id | 对象ID |
| action | 操作类型 |
| actor | 操作人 |
| note | 操作说明 |
| snapshot_before | 操作前快照 |
| snapshot_after | 操作后快照 |
| created_at | 操作时间 |

## 13. 实现优先级

### 第一优先级

- 去掉“成员”作为权限角色。
- 明确子任务负责人、项目负责人、统筹人、CEO、技术管理员。
- 子任务负责人提交状态变更必须进入AI确认中心。
- AI给状态建议，人工可改草稿。
- 项目负责人确认后才写入工作推进表。

### 第二优先级

- 子任务全部完成后提醒负责人关闭关键任务。
- 关键任务已完成后新增子任务，自动重新打开为进行中。
- 删除、批量删除、回收站查看、恢复只给项目负责人和技术管理员。

### 第三优先级

- 项目状态由关键任务汇总。
- 项目完成需要负责人确认。
- 操作日志补全。
- 后端抽出 policy / workflow / projection 服务。

## 14. 设计结论

本系统的核心不是“让AI自动管理项目”，而是：

```text
AI负责提取和建议，
执行人负责提交事实，
负责人负责确认事实，
工作推进表保存确认后的事实。
```

因此，工作推进表的正式逻辑应坚持：

- 子任务是执行单元。
- 关键任务是里程碑。
- 项目是目标容器。
- AI确认中心是事实入库闸口。
- 状态变化必须有责任人和可追溯记录。
