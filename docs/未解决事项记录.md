# 未解决事项记录

本文档用于记录当前系统中已经识别、但尚未完成修复或尚未最终确认的事项。后续开发时可按优先级逐项处理。

## 1. 浏览器缩放与响应式一致性

### 问题

同一个“成果库”页面在 Chrome 和 Edge 中显示比例差异较大。Chrome 中侧边栏、卡片、字体、按钮整体更大；Edge 中整体更紧凑。

### 初步判断

当前更像是浏览器页面缩放比例、`devicePixelRatio` 或视口宽度不同导致的显示差异，不一定是业务代码错误。

### 待验证

- 在 Chrome 和 Edge 中分别执行 `Ctrl + 0`，恢复 100% 缩放后刷新页面。
- 在控制台分别检查：

```js
window.devicePixelRatio
document.documentElement.clientWidth
visualViewport?.scale
```

### 建议解决方案

可以参考以下方向，但需要按实际页面需求调整：

- 关键布局减少完全固定 `px` 宽度。
- 侧边栏、右侧详情栏、卡片区域优先使用 `flex`、`minmax()`、`clamp()` 等响应式约束。
- 成果库、工作推进表、AI确认中心等核心页面统一做一轮跨浏览器视觉验收。

## 2. 成果地址补录通知

### 问题

成果来源有两种：

- 语音更新中 AI 识别出成果，提交人确认后进入负责人审核。
- 成果库中用户单独登记成果，进入成果库内部确认流程。

当前已经确认的产品方向是：成果可以先入库，即使暂时没有存储地址，也允许作为正式成果存在；但后续需要提醒提交人补录成果地址。

### 原因

成果库当前只登记元数据和存储地址，不负责上传真实文件。文件实际保存在知识库、飞书、腾讯文档或其他外部系统中。

### 建议解决方案

可以参考以下方向，但需要按需求实际改动：

- 成果正式入库后，如果 `storage_url` 为空，生成一条“待补录成果地址”的提醒。
- 提醒对象优先为成果提交人。
- 提交人进入成果库时，可以看到“我需要补录地址的成果”。
- 补录地址后，不需要重新走负责人确认，除非后续要求地址变更也需要审核。

## 3. 问题与决策优先级规则

### 问题

“问题与决策”页面中的优先级目前还没有完全钉死规则。用户会看到高、中、低，但不一定清楚系统是如何判断的。

### 当前理解

优先级不应该只是展示字段，而应该反映业务影响程度：

- 是否影响关键任务推进。
- 是否阻塞子任务交付。
- 是否需要统筹人协调。
- 是否需要负责人或 CEO 做决策。
- 是否临近计划时间或已经延期。

### 建议解决方案

可以参考以下方向，但需要按需求实际改动：

- 建立明确的优先级规则表。
- AI 提取时只能给出“建议优先级”，最终可由提交人或负责人修改。
- 负责人确认入库后，问题优先级作为正式字段进入问题库。
- 对“需决策”“高风险”“阻塞关键任务”的事项默认提高优先级。

## 4. 成员侧与负责人侧 UI 一致性

### 问题

部分成员侧页面与负责人侧页面视觉风格不一致，例如背景色、卡片阴影、按钮主色、页面密度存在差异。

### 已处理

已先处理“我的工作台”页面的部分视觉基线：

- 页面背景对齐负责人侧常用底色。
- 统计卡和任务卡增加轻阴影。
- 进行中主色和主按钮渐变调整为负责人侧风格。

### 仍待处理

- 需要继续检查其他成员侧页面是否还存在旧视觉风格。
- 需要确认成果库、问题与决策、AI确认中心、语音更新在不同角色下是否保持一致。
- 需要避免只改颜色，不改布局密度和信息层级。

### 建议解决方案

可以参考以下方向，但需要按需求实际改动：

- 以负责人侧 UI 作为当前项目的视觉基线。
- 抽取统一的页面背景、卡片、按钮、状态徽章样式。
- 优先改动用户高频页面：我的工作台、语音更新、AI确认中心、工作推进表、成果库、问题与决策。

## 5. AI确认中心与语音更新界面继续打磨

### 问题

AI确认中心和语音更新页面已经多次重构，但仍有部分信息层级需要继续收敛：

- 左右区域比例需要持续优化。
- 提取结果与工作推进表三层树状归属要更直观。
- 冗余字段需要继续减少。
- 不同状态下的底部操作区要保持清晰。

### 建议解决方案

可以参考以下方向，但需要按需求实际改动：

- 语音更新：左侧专注输入和模型选择，右侧专注“匹配归属 + 提取结果 + 提交确认”。
- AI确认中心：左侧列表保持紧凑，右侧详情突出三层归属、提取内容、入库去向和操作按钮。
- 已入库、已退回、待确认记录应有明显但不过度占空间的状态提示。

## 6. 正式上线前跨角色回归

### 问题

系统已经涉及多个角色和多条业务闭环，单角色测试不能覆盖全部问题。

### 需要覆盖的角色

- 普通成员
- 项目负责人
- 统筹人
- 项目 CEO
- 技术管理员

### 需要覆盖的核心闭环

- 语音/文字更新 → AI 提取 → 提交负责人 → 确认入库
- 子任务状态变更 → 负责人确认 → 工作推进表更新
- 成果登记 → 成果确认 → 成果库展示
- 问题/风险/待协调/需决策 → 处理/上报/关闭
- 多项目用户切换项目与角色视角

### 建议解决方案

可以参考以下方向，但需要按需求实际改动：

- 将现有自动化测试继续按业务编号整理。
- 每次大改后至少跑核心回归测试。
- 对浏览器视觉差异保留人工验收项。
