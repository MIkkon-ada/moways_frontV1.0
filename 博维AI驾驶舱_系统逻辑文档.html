登录机制、全局角色、项目角色
登录流程
POST /api/auth/login → 比对 passwords.json (SHA256) → 写 session cookie → 中间件校验每次请求
session 存于内存 auth_sessions 表，默认 TTL 由 settings 配置；过期 session 在 startup 时清理。未登录 API 请求返回 401，页面请求重定向到 /login。

项目内角色（project_members.role）
DB 存储键展示名核心权限
owner项目负责人审核本专项提交、确认入库、查看本专项所有内容
coordinator统筹人接收流转至统筹人的提交、反馈意见、转交 CEO
member协同成员提交更新、查看本专项任务和成果
process_guard过程保障同全局过程保障，项目级绑定
project_ceo专项CEO专项内 CEO 角色，批示该专项的待决策事项
角色优先级：owner(4) > coordinator(3) > member/process_guard(2) > project_ceo(1)。同一人可在不同专项有不同角色。

Project / ProjectMember 模块
专项 (Project)

每个专项对应一个 project 记录，所有任务、成果、问题都归属某个 project_id。

默认专项（permissions.py 初始化）

  • 知识资产AI化
  • 顾问作业AI化
  • 交付流程AI化
  • 咨询服务产品化
  • 技术底座与平台预研
成员管理

成员通过 project_members 表绑定到专项，一人可加入多个专项，且每个专项内角色可不同。

子任务（SubTask）
内联展开树 + /api/subtasks/*
展开逻辑
点击关键任务行（或左侧箭头） → toggleInlineSubTasks() → fetchSubTasks(taskId) → 内联插入子任务行
子任务行采用 flatMap 展开，每个主任务返回 [taskRow, ...subRows]；rowSpan 动态计算：展开且无子任务时 +1（显示"暂无子任务"占位行），有 N 条子任务时 +N。

AI 确认中心
ConfirmPage + /api/confirmations/* — 多角色流转状态机
confirm_status 状态分类

待审核 Tab（owner 处理队列）

待确认 待负责人审核 提交人已确认 已重新提交 统筹人已反馈 CEO已批示

流转中 Tab（中间状态）

需修改 已打回提交人 已撤回 已转交统筹人 待CEO决策

已完成 Tab（终态）

已确认 已退回 已确认入库 已入库 不入库 已归档
流转路径（主要分支）
待确认 Owner 审核通过 → 已确认入库
待确认 Owner 打回 → 已打回提交人 提交人修改重提 → 已重新提交
待确认 转交统筹人 → 已转交统筹人 统筹人反馈 → 统筹人已反馈 Owner 最终决定 → 已确认入库
待确认 上报 CEO → 待CEO决策 CEO 批示 → CEO已批示 Owner 确认 → 已确认入库
确认入库后，AI 结果会同步写入 Task 表（更新 status/problem_note 等字段）或 Achievement 表（创建成果记录）。
各角色能访问的内容范围
角色可见队列可执行操作
owner本专项提交（待审核、流转中、已完成）确认/打回/转交统筹人/上报CEO
coordinator被转交给自己的提交反馈意见、转交CEO
CEO / 超级管理员全局所有提交批示、查看所有
过程保障全局所有提交分配、查看（不能自行入库）
普通成员仅自己提交的撤回、修改重提

AI 提取内容结构

AI 从会议记录文本中结构化提取：

  • 任务列表：负责人 + 事项 + 截止时间
  • 决策事项：决策内容 + 决策人
  • 风险事项：风险描述 + 优先级
  • 会议摘要：关键讨论内容概要
提取结果存为 JSON 字段，前端解析展示，可人工修改后发布。
