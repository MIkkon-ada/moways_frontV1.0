## 一、后端改动

### 1. 任务排序逻辑修正（`app/routers/tasks.py`）

**改动：** 将 `GET /api/tasks` 的排序从 `id.asc()` 改为 `created_at.asc()`

**原因：** Excel导入的任务只是规划大纲，ID顺序反映的是Excel行号，不代表实际工作推进顺序。工作推进表应按任务的提交时间升序排列，Excel任务和手工新增任务地位平等。

```python
# 修改前
q.order_by(models.Task.id.asc())

# 修改后
q.order_by(models.Task.created_at.asc())
```

## 二、前端改动（`app/static/app.js`）

### 3. PROJECTS 动态 getter 修复

**改动：** 在 `projectAreas` 数组声明之后，用 `Object.defineProperty` 把 `PROJECTS` 注册为 `window` 上的动态 getter。

**原因：** 重构 `projectAreas` 后，`PROJECTS` 作为全局变量未定义，导致所有引用报错。`components.js` 里的 `const PROJECTS` 是脚本作用域变量，不挂在 `window` 上，不冲突。

```js
Object.defineProperty(window, "PROJECTS", {
  get() { return projectAreas.map(a => a.name); },
  configurable: true
});
```

### 7. 缓存层（`fetchCached` + `invalidate`）

**改动：** 在 `state` 对象后新增轻量缓存机制，解决各页面重复拉取相同数据的问题。

```js
const _cache = {};
const _cacheTs = {};
const _CACHE_TTL = 60_000;   // 60秒过期

function invalidate(...keys) { /* 删除对应缓存条目 */ }
async function fetchCached(key, url) { /* TTL内命中缓存，否则重新请求 */ }
```

**替换 API 调用的位置（共7处）：**

| 函数 | 原调用 | 改为 |
|------|--------|------|
| `fetchAll()` | `api("/api/tasks")` 等4个 | `fetchCached("tasks", ...)` 等4个 |
| `renderTasks()` | `api("/api/tasks")` | `fetchCached("tasks", ...)` |
| `renderAchievements()` | `api("/api/achievements")` | `fetchCached("achievements", ...)` |
| `renderIssues()` | `api("/api/issues")` | `fetchCached("issues", ...)` |
| `renderPeople()` | `api("/api/tasks")` | `fetchCached("tasks", ...)` ← 与renderTasks共用同一缓存 |

**写操作后的缓存失效（共7处）：**

| 操作 | 失效键 |
|------|--------|
| 角色切换（`initRoleSwitch`） | tasks, achievements, issues, confirmations |
| `submitPreview` — 提交更新 | confirmations |
| `confirmWrite` — 确认写入 | tasks, achievements, issues, confirmations |
| `rejectConfirmation` — 退回 | confirmations |
| `transferToYuanjinyu` — 转交袁金玉 | confirmations |
| `confirmAssign` — 分配责任人 | confirmations |
| `saveIssueDetail` — 保存问题 | issues |

**实际效果：**
- 驾驶舱 → 工作推进表：tasks 命中缓存，零额外请求
- 工作推进表 → 组织与分工：tasks 共享缓存，不重复拉取
- 确认写入后，驾驶舱和工作推进表下次加载自动显示最新数据
- 切换身份时彻底清缓存，避免不同用户数据串用

## 八、LLM 提取引擎接入（`app/services/extractor.py`）

### 设计

| 项目 | 说明 |
|------|------|
| 开关 | 环境变量 `BOWEI_USE_LLM=true` 启用，默认 `false`（不影响现有部署） |
| 提供商 | `LLM_PROVIDER` 可选：`anthropic`（默认）/ `dashscope` / `deepseek` / `glm` |
| API Key | anthropic → `ANTHROPIC_API_KEY`；dashscope → `DASHSCOPE_API_KEY`；deepseek → `DEEPSEEK_API_KEY`；glm → `ZHIPUAI_API_KEY` |
| 模型 | claude-sonnet-4-6 / qwen-plus / deepseek-chat / glm-4-flash |
| 回退 | LLM 调用失败（网络/限流/API Key 未配置）自动回退规则引擎，不中断服务 |
| 置信度 | LLM 成功提取时 `confidence=0.93`，规则引擎保持 `0.82/0.58` |

### Prompt 设计

中文指令，严格要求 JSON 输出，包含：
- 合法专项枚举（5个专项）
- 合法 achievement_type 枚举（10种）
- 合法 issue_type、status、priority 枚举
- "不确定时填空字符串或空数组"的明确指示，避免幻觉

### 代码结构

```
extract_update()          ← 对外唯一入口（不变）
  ├── USE_LLM=true → _extract_with_llm()
  │     ├── anthropic → _call_anthropic()    claude-sonnet-4-6
  │     ├── dashscope → _call_dashscope()    qwen-plus（openai兼容接口）
  │     └── 失败 → return None
  │   None → 规则引擎回退
  │   成功 → _normalize_llm_result()         补全缺失字段
  └── USE_LLM=false → _rule_extract()        原规则引擎（保留完整）
```

### 依赖

`requirements.txt` 新增：
- `anthropic>=0.40,<1.0`
- `openai>=1.50,<2.0`（用于 dashscope 兼容接口）

### 启用方式

```bash

# 四选一：
export ANTHROPIC_API_KEY=sk-ant-xxxxx          # Claude（默认）

export LLM_PROVIDER=dashscope                  # 通义千问

export DEEPSEEK_API_KEY=sk-xxxxx
```

### 代码说明

dashscope / deepseek / glm 三者均走 `_call_openai_compat()` 同一个函数，只有 `base_url` / `api_key` / `model` 不同，后续新增其他兼容接口只需加几行参数。

### 5. 系统设置页（`app/static/app.js` — `renderSettings()`，仅管理员可见）

管理员侧边栏新增"系统设置"标签，内容：

- 每个提供商一张卡片：显示名称、当前配置状态（Key 已填 / 未填）
- 点击"配置"按钮弹出抽屉，包含：
  - API Key 输入框（已有 Key 显示 `***` 掩码，清空再输入可替换）
  - Base URL 输入框（已有默认值）
  - 模型名称输入框
  - "启用此提供商"勾选框
  - "测试连接"按钮（**直接用表单当前值测试**，无需先保存）
  - "保存"按钮

**Bug 修复：** 早期实现将整个 JSON 对象序列化后内联到 `onclick` HTML 属性，特殊字符导致 HTML 解析失败。改为将配置对象存入 `_llmConfigs` 模块变量，`onclick` 只传 `provider` 字符串，弹窗内从 `_llmConfigs[provider]` 读取。

样式类：`.settings-wrap`、`.settings-header`、`.model-config-list`、`.model-config-card`、`.model-config-head`、`.model-config-info`、`.inline-check`、`.modal-actions`
