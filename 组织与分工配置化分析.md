# 组织与分工配置化分析

## 1. 当前问题

当前「组织与分工」页面中的 `projectAreas` 和 `members` 是写在前端 `app.js` 里的静态 mock 数据。

这种方式适合快速验证页面结构，但不适合长期使用，也不适合给其他小组或其他公司复用。

主要问题有三类：

1. **换一个项目就要改代码**
   - 专项名称、负责人、确认人、协同成员都写死在 JS 文件里。
   - 如果换成别的小组，需要开发人员手动改代码。

2. **数据和业务规则混在一起**
   - 页面展示逻辑、权限规则、专项配置混在同一个前端文件里。
   - 后续 AI 确认中心要判断“谁能确认什么”时，不应该依赖前端写死的数据。

3. **无法形成统一导入模板**
   - 当前 Excel 已经有组织与分工信息，但前端没有直接读取配置结果。
   - 后续给别的小组使用时，最好只替换配置表，而不是改代码。

## 2. 哪些内容不应该写死

以下内容都应该配置化，而不是写死在前端：

### 专项配置

```ts
{
  id: string
  name: string
  coordinator: string
  owner: string
  confirmers: string[]
  collaborators: string[]
  taskCount: number
  riskCount: number
}
```

其中真正应该配置的是：

- 专项名称
- 统筹人
- 负责人
- 确认人
- 协同成员

`taskCount` 和 `riskCount` 不应该配置，应由任务表、问题表自动统计。

### 人员配置

```ts
{
  id: string
  name: string
  role: string
  responsibleArea: string
  responsibility: string
  confirmScope: string
  viewScope: string
}
```

其中应配置的是：

- 姓名
- 项目角色
- 负责专项
- 责任内容
- 可确认内容
- 查看范围

项目角色建议固定为枚举：

- 组长
- 专项统筹人
- 专项负责人
- 协同成员
- 过程保障
- AI应用工程师

查看范围建议固定为枚举：

- 全部
- 本专项
- 本人相关
- 技术相关

## 3. 推荐的数据来源

建议不要让前端维护组织与分工数据，而是采用以下来源之一。

### 方案一：继续使用 Excel 导入

适合当前阶段。

在 Excel 中维护：

1. 组织与分工表
2. 专项责任分工表
3. 人员职责与权限表

导入后写入 SQLite，前端从接口读取。

优点：

- 适合非技术人员维护。
- 换公司、换小组时只需要替换 Excel。
- 和当前项目已有的工作推进总表机制一致。

缺点：

- 需要约定 Excel 模板格式。
- 需要导入前校验字段。

### 方案二：使用 JSON 配置文件

适合轻量部署。

例如：

```json
{
  "projectAreas": [],
  "members": []
}
```

优点：

- 比写死在 JS 中更清晰。
- 适合版本管理。
- 不依赖数据库。

缺点：

- 非技术人员维护不方便。
- 仍然需要重启或刷新配置。

### 方案三：做后台配置页面

适合后续正式产品化。

增加组织与权限配置页面，支持维护：

- 专项
- 人员
- 确认人
- 查看范围
- 确认范围

优点：

- 最适合长期复用。
- 可以直接支撑权限判断。

缺点：

- 当前阶段成本较高。
- 容易把系统做成后台管理系统，需要控制设计边界。

## 4. 推荐演进路径

建议按三个阶段演进。

### 阶段一：从前端 mock 改为 Excel 导入

目标：

- 去掉 `app.js` 中写死的 `projectAreas` 和 `members`。
- 从 SQLite 中读取组织分工数据。
- 保持页面展示不变。

需要新增或调整的数据表：

#### project_areas

| 字段 | 说明 |
|---|---|
| id | 专项ID |
| name | 专项名称 |
| coordinator | 统筹人 |
| owner | 负责人 |
| confirmers | 确认人，多人用 JSON 或分隔符存储 |
| collaborators | 协同成员 |

#### member_permissions

| 字段 | 说明 |
|---|---|
| id | 人员ID |
| name | 姓名 |
| role | 项目角色 |
| responsible_area | 负责专项 |
| responsibility | 责任内容 |
| confirm_scope | 可确认内容 |
| view_scope | 查看范围 |

### 阶段二：确认中心使用确认人规则

目标：

- AI 提取出专项后，系统知道应该进入谁的确认队列。
- 例如识别为“知识资产AI化”，确认人就是“杨宇帆、刘万超”。

需要实现：

- 根据专项匹配 `project_areas.confirmers`
- 确认中心按当前身份过滤待确认建议
- 组长或过程保障可查看全部待确认事项

### 阶段三：支持多项目/多公司配置

目标：

- 同一套系统可服务不同项目或公司。
- 每个项目有独立的专项、人员、权限、任务、成果。

需要增加：

- project_id
- company_id 或 tenant_id
- 每张业务表关联项目ID

## 5. 给其他小组使用前需要准备什么

如果下次给别的小组使用，至少要准备以下信息：

### 专项分工

- 有哪些专项
- 每个专项的统筹人
- 每个专项的负责人
- 每个专项的确认人
- 每个专项的协同成员

### 人员权限

- 每个人的姓名
- 项目角色
- 负责专项
- 责任内容
- 可确认内容
- 查看范围

### 任务主数据

- 初始工作推进表
- 每条任务所属专项
- 负责人
- 计划时间
- 关键成果
- 完成标准

### 成果库规则

- 成果类型
- 复用标签
- 是否必须关联任务
- 是否需要版本号

### AI识别词典

- 专项别名
- 人名别名
- 常见任务关键词
- 常见成果关键词
- 风险和决策关键词

## 6. 当前建议

短期建议：

- 继续保留当前页面展示。
- 但不要再往 `app.js` 里增加写死数据。
- 下一步应把 `projectAreas` 和 `members` 改为从后端接口读取。

中期建议：

- 将 Excel 的“组织与分工”解析结果写入 SQLite。
- 前端只负责展示，不负责维护组织规则。

长期建议：

- 增加项目级配置能力。
- 让不同小组通过导入配置表完成初始化，而不是修改代码。

## 7. 判断标准

组织与分工配置化完成后，应能回答：

1. 换一个小组，是否只需要换配置表？
2. AI 识别出某个专项后，是否能自动知道给谁确认？
3. 当前登录人是否能根据权限只看到自己该看的内容？
4. 新增任务、成果、问题是否能自动落到正确专项？

如果这些问题还需要改代码才能解决，说明配置化还没有完成。
