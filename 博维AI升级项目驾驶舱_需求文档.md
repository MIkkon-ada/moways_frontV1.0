# 博维 AI 升级项目驾驶舱需求文档

版本：V1.0  
日期：2026-06-01  
状态：需求评审稿  
项目类型：企业内部 AI 升级项目可视化驾驶舱

## 1. 背景

博维 AI 升级项目涉及多个并行推进的专题，包括任务推进、成果沉淀、风险跟踪、会议纪要、语音更新、AI 确认、统筹建议、决策事项、项目管理和成员管理等内容。当前信息分散在多个页面和流程中，管理者需要在不同视图之间切换才能了解项目整体状态，导致信息获取成本高、协同效率低、决策反馈慢。

本项目拟建设一套“升级项目驾驶舱”，把项目进展、风险、成果、待决策事项和 AI 确认内容集中到统一门户中，帮助管理者快速掌握全局，帮助项目成员按项目维度推进工作，并保持与现有后端数据结构和权限体系兼容。

## 2. 目标

1. 让管理者在首页即可快速看到当前登录人可见范围内的全部项目概览。
2. 让项目负责人和成员围绕具体项目推进任务、提交成果、关闭问题并形成闭环。
3. 让 AI 提取结果进入人工确认中心，确认后再写入正式业务数据。
4. 在保留现有 React 前端和后端 API 结构的前提下，尽量复刻旧版驾驶舱的成熟视觉风格。
5. 所有正式业务数据必须来源于后端真实接口，不新增 mock 数据，不写死演示数据。

## 3. 适用范围

本需求文档适用于博维 AI 升级项目驾驶舱的前端重构和功能完善，重点覆盖以下页面和能力：

1. 首页驾驶舱
2. 语音更新
3. 会议纪要
4. AI 确认中心
5. 工作推进
6. 成果库
7. 问题与决策
8. 统筹建议
9. 决策事项
10. 项目管理
11. 成员管理

## 4. 非目标

1. 不恢复旧版 `app/static` 技术栈。
2. 不把旧版 `app.js`、`dashboard.js` 直接搬进 React。
3. 不新增 mock 接口或演示数据来替代真实数据。
4. 不在截图中伪造真实未存在的数据。
5. 不为 AI 结果绕过人工确认直接写入正式业务数据。
6. 在真实 Linux 服务器验证未完成前，不声称正式上线 ready。

## 5. 用户角色

| 角色 | 核心诉求 | 典型能力 |
|---|---|---|
| super_admin | 全局配置、全局查看、系统管理 | 查看全部项目、管理项目、管理成员 |
| project_ceo | 关注重点风险和决策事项 | 查看决策事项、审批关键问题 |
| coordinator | 统筹多个专项推进 | 查看统筹建议、协调跨项目事项 |
| owner | 负责单个专项推进 | 管理本项目任务、成果、问题 |
| member | 执行任务和提交进展 | 查看参与项目、提交更新、跟进任务 |
| process_guard | 过程保障和质量把关 | 查看过程风险、确认流程合规 |

说明：当前数据库中的部分角色命名可能与产品预期不完全一致，例如“技术支持者”是否应具备全局可见范围，需要后续做一次角色数据校准。

## 6. 产品原则

1. 首页默认展示当前登录人可见范围内的全部项目概览。
2. 用户可通过顶部筛选器切换到某个具体项目，但不应强制收窄首页默认范围。
3. 具体业务页仍保持项目维度，继续按 `project_id` 过滤。
4. 所有页面必须遵守当前权限体系，不得通过前端隐藏替代后端控制。
5. AI 生成内容必须经过人工确认后才进入正式业务数据。
6. 数据治理、备份恢复、部署验证都应以防污染为前提。

## 7. 信息架构

当前前端菜单建议按旧版驾驶舱口径展示：

1. 首页驾驶舱
2. 语音更新
3. 会议纪要
4. AI 确认中心
5. 工作推进
6. 成果库
7. 问题与决策
8. 统筹建议
9. 决策事项
10. 项目管理
11. 成员管理

路由结构保持不变，不因展示名称变化而修改现有路径设计。

## 8. 首页驾驶舱需求

### 8.1 默认范围

首页默认显示“全部可见项目”，即当前登录人有权限访问的所有项目。用户可以在首页通过筛选条件切换到某一个具体项目。

### 8.2 数据来源

首页必须使用现有 dashboard overview 接口：

1. 全部可见范围：`GET /api/dashboard/overview`
2. 单项目范围：`GET /api/dashboard/overview?project_id=<id>`

禁止新增 mock 数据和写死演示数据。

### 8.3 页面结构

首页视觉应尽量复刻旧版驾驶舱模板，建议包含：

1. 顶部标题区
2. 副标题区，说明当前可见项目的整体进展、风险、成果和待决策事项
3. 筛选行，包含项目、负责人、状态、月份
4. KPI 六卡，展示任务总数、进行中、已完成、延期、需决策事项、成果数
5. 第一行双栏，展示项目总览和任务状态分布
6. 第二行三栏，展示本周重点、延期任务、需决策事项
7. 第三行双栏，展示主要风险和最新成果

### 8.4 首页验收

1. 登录后进入首页不白屏。
2. `/project/:id` 首页默认仍显示全部可见项目概览。
3. 首页筛选选择某项目后，overview 请求必须带 `project_id`。
4. 首页切回全部可见项目后，overview 请求不带 `project_id`。
5. 其他项目业务页仍按项目维度工作。
6. 菜单权限不变化。

## 9. 全局界面要求

### 9.1 Topbar

1. 白色横向顶栏，贯穿全屏。
2. 高度约 56px。
3. 左侧显示 Logo 和“博维 AI 升级项目驾驶舱”。
4. 中部显示搜索框，搜索对象应至少包含项目、任务、文档、会议等。
5. 右侧显示月份、通知、用户头像、用户名。
6. 顶栏下方使用轻分割线，不做大圆角悬浮卡片风格。

### 9.2 Sidebar

1. 浅色背景，宽度约 220px。
2. 从顶栏下方开始布局。
3. 直接展示菜单项，不额外增加大块“项目导航说明区”。
4. active 菜单使用蓝色渐变或高亮态。
5. 菜单项高度约 42px，图标与文字左对齐。
6. 底部可保留收起和用户信息，但不能抢主导航视觉。

### 9.3 内容区

1. 内容背景为浅灰蓝或浅灰白。
2. 卡片白底、细边框、轻阴影。
3. 页面 padding 控制在 20px 到 24px 左右。
4. 卡片间距控制在 12px 到 16px 左右。
5. 信息密度应接近旧版驾驶舱，不做过于松散的后台风格。

## 10. 工作推进页需求

工作推进页用于管理任务台账，后续 UI 复刻应对齐旧版逻辑：

1. 顶部标题和筛选条。
2. 状态 KPI，至少包含任务总数、进行中、已完成、延期、待确认。
3. 主表格展示任务名称、所属项目、负责人、状态、截止日期、更新时间。
4. 支持查看任务详情。
5. 保留当前权限和项目过滤逻辑。
6. 不新增后端 API，除非进入明确的标准化改造批次。

## 11. 成果库需求

成果库用于沉淀已确认成果：

1. 成果统计卡。
2. 成果列表或卡片网格。
3. 展示成果名称、类型、负责人、关联项目、更新时间。
4. 支持空态。
5. 不使用 mock 成果。

## 12. 问题与决策需求

问题与决策页面用于跟踪风险、问题和闭环：

1. 问题统计卡，至少包含未关闭、高风险、待决策、已闭环。
2. 问题清单表格。
3. 决策事项面板。
4. 问题闭环追踪。
5. 风险等级颜色语义：
   - 红色：高风险
   - 橙色：中风险
   - 蓝色：进行中
   - 绿色：已闭环

## 13. 语音更新需求

语音更新页面用于提交进展材料：

1. 支持录音、上传、粘贴文本等入口。
2. 展示 AI 提取结果。
3. 支持关联项目和任务。
4. 提交后进入确认流程。
5. 不可直接绕过确认写入正式数据。

## 14. 会议纪要需求

会议纪要页面用于沉淀会议内容并提取任务：

1. 会议正文区域。
2. 自动识别任务清单。
3. 待校对事项。
4. 历史会议记录。
5. 与确认中心流程联动。

## 15. AI 确认中心需求

AI 确认中心用于人工确认 AI 提取内容：

1. 待确认队列。
2. 当前记录详情。
3. 入库方向。
4. 操作包括确认写入、退回修改、标记补充。
5. 操作日志。
6. 角色权限保持与后端一致。

## 16. 项目管理与成员管理需求

### 16.1 项目管理

1. 项目列表。
2. 项目状态。
3. 归档信息。
4. 项目详情。
5. 维持与现有业务树的项目层级关系。

### 16.2 成员管理

1. 成员台账。
2. 角色配置。
3. 组织分工说明。
4. 后续可逐步对齐旧版“组织与分工 / 系统设置”视觉入口。

## 17. 数据与权限要求

1. 所有业务数据必须来自后端真实 API。
2. 首页全部范围必须限制在当前登录人可见项目内。
3. 单项目页面必须继续按 `project_id` 过滤。
4. 不得通过前端隐藏来替代后端权限控制。
5. 不同角色看到的菜单和页面能力必须继续由权限配置控制。

## 18. 部署与上线约束

当前状态：

1. 后端 L2 生产化 baseline 已完成。
2. 数据库 L3 readiness baseline 已完成。
3. 同域部署配置已完成。
4. 服务器预装包和上线验收清单已完成。
5. `L4-2S` 真实服务器验证仍然 blocked。

上线前必须完成：

1. 真实 Linux 服务器部署。
2. `nginx -t`。
3. `systemd-analyze verify`。
4. `/api/health` 服务器直连和同域反代验证。
5. HTTPS Cookie Secure 验证。
6. 浏览器深链刷新验证。
7. UAT 验收。

没有完成真实服务器验证前，不得声明正式上线 ready，也不得打正式上线标签。

## 19. 迭代优先级

1. 第一批：首页范围修正，默认显示全部可见项目。
2. 第二批：全局壳层和首页驾驶舱继续按旧版截图复刻。
3. 第三批：工作推进页旧版 AI 复刻。
4. 第四批：成果库、问题与决策旧版 UI 复刻。
5. 第五批：语音更新、会议纪要旧版 AI 复刻。
6. 第六批：AI 确认中心旧版 UI 复刻。
7. 第七批：组织与分工、项目管理、成员管理视觉收口。
8. 第八批：真实服务器部署验证和 UAT。

## 20. 验收标准

### 20.1 功能验收

1. 登录、退出、会话保持正常。
2. 首页默认显示全部可见项目。
3. 首页支持按项目筛选。
4. 项目页、任务页、成果页、问题页、会议页、提交页、确认中心可正常进入。
5. 权限菜单不被破坏。
6. 项目切换正常。

### 20.2 UI 验收

1. Topbar、Sidebar、首页布局尽量贴近旧版截图。
2. 首页 KPI、进度、状态分布、重点事项、风险、成果模块完整。
3. 页面无白屏、无错位、无大面积空白或文字挤压。
4. 不出现与旧版风格冲突的夸张 hero 卡片、深色后台风格或自由设计风格。

### 20.3 技术验收

1. `npm run build` 通过。
2. 后端基础联调通过。
3. 不引入 `dist/`、数据库、日志、真实 secret 到交付包。
4. 不修改后端 API 和数据库结构，除非进入明确的后端改造批次。

## 21. 当前风险

1. 旧版 UI 复刻和首页范围修正容易混在同一批，必须严格分批提交。
2. 当前部分成员角色和产品预期可能不一致，需要单独做角色数据校准。
3. 真实服务器验证尚未完成，部署链路不能算正式通过。
4. SQLite 不适合多机部署，正式扩展前需要重新评估数据库方案。
5. UI 复刻时不能牺牲当前 React 数据流、权限和路由稳定性。

## 22. 下一步建议

1. 先确认首页“全部可见项目”逻辑是否符合产品预期。
2. 再恢复已暂存的 UI 复刻改动，继续对齐旧版截图。
3. 每个 UI 批次先完成截图确认，再提交。
4. 拿到真实服务器后执行 `L4-2S` 验证。
5. UAT 通过后再进入正式上线判断。
