# AI项目驾驶舱新公司适配准备清单

本文档用于在将“AI升级项目驾驶舱”迁移或复用到其他公司、其他项目之前，提前收集基础配置数据。

系统能否顺利跑通，不取决于页面是否改名，而取决于是否提前准备好专项、人员、任务、成果、权限和AI识别规则。

## 1. 项目基础信息

需要先明确项目的基本边界。

| 信息项 | 说明 | 示例 |
|---|---|---|
| 项目名称 | 当前驾驶舱服务的项目名称 | XX集团AI应用落地项目 |
| 项目周期 | 项目起止时间或阶段周期 | 2026年1月-2026年6月 |
| 项目阶段 | 当前所处阶段 | 启动期 / 试点期 / 推广期 |
| 项目总负责人 | 对项目结果负责的人 | 张三 |
| 项目目标 | 本项目要达成的业务目标 | 建立AI赋能销售、交付、知识管理的工作体系 |
| 汇报对象 | 需要重点查看驾驶舱的人 | CEO / 项目委员会 / 数字化负责人 |

## 2. 专项或工作模块

系统需要知道项目被拆成哪些专项。每个专项会影响首页驾驶舱、AI任务归类、确认中心分发和权限判断。

建议准备字段：

| 字段 | 说明 |
|---|---|
| 专项名称 | 固定工作模块名称 |
| 专项目标 | 该专项要解决什么问题 |
| 统筹人 | 对专项推进节奏负责的人 |
| 负责人 | 对专项具体交付负责的人 |
| 确认人 | 有权限确认AI提取结果并写入系统的人 |
| 协同成员 | 参与该专项的人 |
| 关键任务范围 | AI识别任务时的归类依据 |
| 成果要求 | 该专项应沉淀哪些成果 |
| 风险关注点 | 首页和问题看板需要关注的风险 |

示例：

| 专项名称 | 统筹人 | 负责人 | 确认人 | 协同成员 |
|---|---|---|---|---|
| 知识资产AI化 | 刘万超 | 杨宇帆 | 杨宇帆、刘万超 | 袁金玉、郭熠彬、吴肖 |

## 3. 人员与权限

人员配置不仅用于展示组织分工，也用于后续AI确认中心判断“谁能确认什么”。

建议准备字段：

| 字段 | 说明 |
|---|---|
| 姓名 | 人员姓名 |
| 项目角色 | 组长 / 专项统筹人 / 专项负责人 / 协同成员 / 过程保障 / AI应用工程师 |
| 所属组织 | 所属部门、团队或项目组 |
| 负责专项 | 该人员主要负责的专项 |
| 责任内容 | 该人员在项目中的职责说明 |
| 可确认内容 | 该人员能确认哪些AI提取结果 |
| 查看范围 | 全部 / 本专项 / 本人相关 / 技术相关 |

项目角色建议统一使用以下枚举：

- 组长
- 专项统筹人
- 专项负责人
- 协同成员
- 过程保障
- AI应用工程师

查看范围建议统一使用以下枚举：

- 全部
- 本专项
- 本人相关
- 技术相关

## 4. 任务主数据

任务主数据是工作推进表、首页驾驶舱、AI任务匹配的基础。

建议准备一张初始工作推进表，字段包括：

| 字段 | 说明 |
|---|---|
| 专项 | 所属专项 |
| 关键任务 | 任务名称 |
| 关键成果 | 任务应产出的成果 |
| 完成标准 | 判断任务完成的标准 |
| 统筹人 | 任务统筹人 |
| 负责人 | 任务负责人 |
| 协同成员 | 参与人员 |
| 计划时间 | 计划完成时间或周期 |
| 当前状态 | 未开始 / 推进中 / 已完成 / 延期 / 风险 |
| 问题与需协调事项 | 当前阻塞或需协调内容 |
| 下周计划 | 下一步计划 |

注意事项：

- 每条任务必须归属到一个专项。
- 每条任务最好有明确负责人。
- 关键成果越清晰，后续AI识别和成果匹配越准确。

## 5. 成果库规则

成果库不是文件列表，而是项目资产库。迁移到新公司前，需要提前定义成果分类和归档规则。

建议准备字段：

| 字段 | 说明 |
|---|---|
| 成果名称 | 成果标题 |
| 成果类型 | 方案、表格、模板、SOP、Prompt、Agent原型、会议纪要、复盘报告、案例包、产品材料 |
| 所属专项 | 成果归属专项 |
| 关联任务 | 成果来自哪条任务 |
| 负责人 | 成果责任人 |
| 版本 | 重要成果需要版本号 |
| 文件链接 | 文件地址或内部链接 |
| 适用场景 | 成果适合在哪些场景复用 |
| 更新日期 | 最近更新时间 |
| 复用标签 | 内部使用 / 项目复用 / 产品材料 / 客户交付 |

建议归档规则：

- 所有阶段性成果必须关联任务。
- 重要成果必须标记版本。
- 客户资料需脱敏后入库。
- AI新增成果如果与预定成果一致，应更新原成果状态；如果不一致，应作为补充成果新增。

## 6. 问题与决策规则

问题与决策规则会影响风险看板和首页“需决策事项”。

建议准备字段：

| 字段 | 说明 |
|---|---|
| 类型 | 问题 / 风险 / 需协调 / 决策事项 |
| 描述 | 问题或决策事项说明 |
| 所属专项 | 关联专项 |
| 责任人 | 负责推动解决的人 |
| 协同人 | 需要配合的人 |
| 优先级 | 高 / 中 / 低 |
| 状态 | 待处理 / 处理中 / 需决策 / 已解决 |
| 需决策人 | 需要谁拍板 |
| 预计解决时间 | 计划解决时间 |
| 解决结论 | 已解决后的结论 |
| 关联任务 | 对应任务 |

需要提前明确：

- 什么情况算“高优先级风险”。
- 什么情况需要上升到项目总负责人。
- 问题关闭的判断标准是什么。

## 7. AI识别规则

AI结构化提取依赖业务词典。换公司使用前，需要补充新公司的语义规则。

建议准备：

| 规则类型 | 示例 |
|---|---|
| 专项别名 | “知识库那块”对应“知识资产AI化” |
| 人名别名 | “海总”对应“冯海林” |
| 任务关键词 | 梳理、试点、验证、复盘、上线 |
| 成果关键词 | 方案、模板、SOP、Prompt、清单、报告 |
| 风险关键词 | 卡住、延期、缺资源、无法推进、需要协调 |
| 决策关键词 | 需要拍板、请确认、需决策、待定 |
| 状态判断规则 | “已经完成初稿”可判断为推进中或已形成初版成果 |

这些规则越贴近业务口语，豆包转写文本进入系统后的识别质量越高。

## 8. 建议准备的导入模板

为了降低迁移成本，建议为每家公司准备以下表格：

1. 组织与分工表
2. 专项责任分工表
3. 工作推进总表
4. 成果类型与归档规则表
5. 问题与决策规则表
6. AI识别词典表

## 9. 最低准备清单

如果时间有限，至少要准备以下内容：

- 项目名称和项目负责人
- 专项清单
- 每个专项的统筹人、负责人、确认人
- 人员角色与查看范围
- 初始任务清单
- 成果类型和复用标签
- 问题类型和决策人规则
- 常见人名、专项名、任务名的口语别名

## 10. 迁移判断标准

完成以上准备后，新公司版本至少应该能回答三个问题：

1. 谁负责哪个专项？
2. AI提取结果应该给谁确认？
3. 这个人能看哪些范围、确认哪些内容？

如果这三个问题回答不清楚，不建议直接上线使用。
