```

├── routers/
│   ├── tasks.py          工作推进表
│   ├── updates.py        提交进度更新
```

**身份识别方式：** 每个请求通过 HTTP Header `X-Current-User` 传入当前用户姓名（中文），后端解码后查权限。没有 token/session，完全信任 Header。

## 三、权限系统（permissions.py）

### 双模式设计

`get_user_context_from_db()` 是主入口：
1. **优先查数据库**：从 `people` 表读 `system_role`，从 `project_memberships` 表读专项角色
2. **回退静态配置**：如果数据库没有该用户记录，自动回退到 `permissions.py` 中硬编码的 `PROJECT_AREAS` 静态映射

这样既支持运行时动态修改权限，又保证初始部署不依赖数据库数据。

### 角色体系

| 角色 | system_role | 视野 | 写权限 | 确认权限 | 分配权限 |
|------|------------|------|--------|----------|----------|
| CEO（冯海林） | ceo | 全部专项 | 全部 | 全部 | 全部 |
| 技术管理员（吴肖/郭熠彬） | tech_admin | 全部专项 | 全部 | 全部 | 全部 |
| 过程管控（袁金玉） | process_guard | 全部专项 | 无 | 无 | 全部 |
| 统筹人（刘万超/邹奇敏） | — | 负责的专项 | 无 | 无 | 无 |
| 负责人（杨宇帆等） | — | 负责+协同专项 | 本专项 | 本专项 | 无 |
| 协同成员 | — | 协同专项（只读） | 无 | 无 | 无 |

### 核心判断函数
```python
can_view_project(context, project)    # 能否查看某专项的数据
can_write_project(context, project)   # 能否写入某专项的数据（CEO/tech_admin/owner）
can_confirm_submission(context, data) # 能否确认某条提交（CEO/tech_admin + 该专项owner）
can_assign_submission(context)        # 能否分配提交（CEO/tech_admin/process_guard）
```

## 四、核心业务流程：提交 → 提取 → 确认 → 写入

这是整个系统最复杂的流程，涉及 `updates.py`、`confirmations.py`、`extractor.py` 三个模块。

```
用户输入文本
    │
    ▼
POST /api/updates/extract        ← 只提取，不入库
    │  extractor.py 关键词分析
    │  返回 suggestion JSON
    ▼
前端展示AI提取结果（确认中心预览）
    │
    ▼
POST /api/updates               ← 正式提交，创建 UpdateSubmission
    │  ai_result_json = suggestion
    │  human_result_json = ai_result_json（初始相同）
    │  confirm_status = "待确认"
    ▼
GET /api/confirmations          ← 列出待处理的提交
    │  Tab "待确认" → confirm_status IN ["待确认"]
    │  Tab "待分配" → confirm_status IN ["需修改"]  ← 注意: 前端"待分配"对应DB"需修改"
    │  Tab "已确认" → confirm_status IN ["已确认","已退回"]
    ▼
负责人查看详情，可在右侧面板修改AI字段
    │
    ├──▶ POST /confirmations/{id}/save
    │        保存人工修改到 human_result_json
    │        confirm_status = "需修改" （进入"待分配"Tab）
    │
    ├──▶ POST /confirmations/{id}/assign
    │        在 human_result_json 里设置 assigned_to
    │        confirm_status = "待确认" （回到"待确认"Tab）
    │
    ├──▶ POST /confirmations/{id}/confirm   ← 最终写入
    │        解析 human_result_json
    │        写入 Task（新建或更新）
    │        写入 Achievement（匹配已有 or 新建）
    │        写入 Issue
    │        confirm_status = "已确认"
    │
    └──▶ POST /confirmations/{id}/reject
             confirm_status = "已退回"
             记录 reject_reason
```

### 确认写入的智能匹配（confirmations.py）

`fulfill_or_create_achievement()` 用 Python 标准库的 `SequenceMatcher` 做成果名称相似度匹配（阈值 0.58）：
- 如果 AI 提取的成果名称与数据库中某个**已计划**成果的名称相似度 ≥ 0.58，则**更新**该已计划成果
- 否则**新建**一条成果记录

这样可以把 AI 自然语言描述的成果自动关联到 Excel 预规划的成果条目上。

## 五、AI提取引擎（services/extractor.py）

当前 `USE_LLM = False`，完全用规则引擎（关键词匹配）。

### 提取逻辑
```
输入文本
  ├── _pick_project()    关键词匹配专项（知识/问答→知识资产AI化，质检→顾问作业AI化，等）
  ├── _split_items()     按语义关键词筛选句子（完成/产出 → completed；计划/下一步 → next_steps）
  ├── _status()          判断任务状态（完成/上线→已完成；延期/卡住→延期；推进/开发→推进中）
  └── _achievement_type()  判断成果类型（SOP/Prompt/Agent原型/模板/方案 等）
```

### 输出结构
```json
{
  "summary": "输入文本前180字",
  "special_project": "识别专项",
  "related_task": "下一步计划（第一条）",
  "completed_items": ["已完成事项列表"],
  "achievements": [{"name","achievement_type","special_project","owner",...}],
  "issues": [{"issue_type","description","priority","need_decision_by",...}],
  "next_steps": ["下一步计划列表"],
  "status_suggestion": "推进中",
  "confidence": 0.82,
  "task": {特别整理的任务字段},
  "meeting": {会议纪要字段（仅meeting类型有）}
}
```

### 当前问题
- `_pick_project()` 在无法匹配时 fallback 到 `"知识资产AI化"`，**误分类风险高**
- `coordinator` 字段硬编码为 `"袁广民"`（非博维团队成员，遗留数据）
- 接入真实 LLM 的扩展点已预留（`if USE_LLM: pass`），但尚未实现

## 六、各资源的标准CRUD模式

`tasks.py`、`achievements.py`、`issues.py` 遵循相同模式：

| 接口 | 说明 |
|------|------|
| `GET /api/{resource}` | 列表查询，自动过滤无权限专项，支持 type/project/owner/status 筛选 |
| `POST /api/{resource}` | 新建，校验写权限，写 OperationLog（"新建xxx"） |
| `GET /api/{resource}/{id}` | 单条查询，校验可见权限 |
| `PUT /api/{resource}/{id}` | 全量更新，校验写权限，写 OperationLog（"修改xxx"） |
| `DELETE /api/{resource}/{id}` | 删除，校验写权限，写 OperationLog（"删除xxx"） |
| `PATCH /api/{resource}/{id}/status` | 仅更新状态，写 OperationLog（"更新xxx状态"） |

### 排序规则
- `tasks`: `created_at ASC`（按提交时间升序，Excel导入任务和手工录入任务地位相同）
- `achievements`: `updated_at DESC`（最近更新的成果排前面）
- `issues`: `updated_at DESC`

## 七、驾驶舱聚合（dashboard.py）

`GET /api/dashboard/overview` 一次性返回所有首页所需数据：

| 字段 | 内容 |
|------|------|
| `summary` | 任务总数/成果总数/未解决问题数/待确认提交数 |
| `project_cards` | 每个专项的完成率、成果数、问题数、最后更新时间 |
| `status_stats` | 各状态的任务数量统计 |
| `decisions` | `need_decision_by="请海总"` 且未解决的问题（前10条） |
| `risks` | `priority="高"` 且未解决的问题（前10条） |
| `pending_tasks` | 未完成任务（前10条） |
| `latest_achievements` | 最近更新的成果（前10条） |

注意：dashboard 接口目前**没有权限过滤**，任何人都能看到全部数据。

## 八、操作日志（logs.py）

`GET /api/logs?target_type=task&target_id=xxx`

- 支持 `task` / `issue` / `achievement` 三种类型
- 对 task/issue/achievement 会验证当前用户对该记录所属专项是否有查看权限
- 其他类型（如 `submission`）不做权限校验，直接返回
- 按 `created_at ASC` 排序（从旧到新）

## 十、数据一致性保障

1. **每次写操作都记 OperationLog**：`crud.log()` 在 `db.flush()` 后、`db.commit()` 前写日志，保证原子性
2. **AI原则**：AI只生成建议（`UpdateSubmission`），不直接覆盖 Task/Achievement/Issue；只有确认后才写入
3. **冲突处理**：确认接口如果找到同名的已计划任务（`related_task_id` 不空），会更新而不是新建
4. **软删除**：Person 表有 `is_active` 字段，人员停用不物理删除
