# 博维AI升级项目驾驶舱 — 前端设计问题清单

> 基于 `app/static/app.js` + `app/static/components.js` 代码审查
> 生成日期：2026-05-24

---

## 一、严重 Bug（功能完全失效）

### Bug-01 确认中心的编辑内容被丢弃

**位置：** `app.js` → `confirmWrite()` / `openConfirmation()` / `confirmField()`

**现象：** 审核人在右侧详情面板修改了 AI 提取的字段（`contenteditable` 可编辑），点"确认写入"后，所有修改全部丢失，写入的仍是 AI 原始数据。

**根因：**
```js
async function confirmWrite(id) {
  const d = await api(`/api/confirmations/${id}`);        // 重新拉服务器原始数据
  await api(`/api/confirmations/${id}/confirm`, {
    body: JSON.stringify({ human_result: d.human_result || d.ai_result }), // 没读DOM
  });
}
```
`confirmWrite()` 完全没有读取详情面板里 `contenteditable` 元素的当前内容，用户的修改等于没有发生。

**修复方向：** 在调用 API 前，用 `readContainer()` 或类似方式读取详情面板内所有 `[data-field]` 元素的当前值，构造修改后的 `human_result` 再提交。

---

### Bug-02 会议纪要"发布"按钮没有任何处理

**位置：** `app.js` → `modeInputArea("meeting")` / `markMeetingReviewed()`

**现象：** 袁金玉点"完成校对"后，"发布"按钮变为可点击，但点击之后什么都不发生。

**根因：**
```html
<button type="button" id="publishMeetingBtn" disabled>发布</button>
```
按钮没有 `onclick` 处理函数。`markMeetingReviewed()` 只是前端解禁了按钮状态，没有写 API 调用。

**修复方向：** 给发布按钮添加 `onclick="publishMeeting()"` 并实现该函数，调用 `/api/updates` 提交会议纪要数据。

---

### Bug-03 `normalizeProject()` 的 fallback 会污染数据

**位置：** `components.js` → `normalizeProject()`

**现象：** 专项字段为空的记录（AI 未识别出专项时）会被强制归入"知识资产AI化"，导致杨宇帆看到大量不属于他的记录。

**根因：**
```js
function normalizeProject(value) {
  return PROJECT_ALIASES[value] || value || "知识资产AI化";  // 空字符串时 fallback 错误
}
```

**修复：**
```js
function normalizeProject(value) {
  return PROJECT_ALIASES[value] || value || "";  // 空值就返回空字符串
}
```

---

### Bug-04 `components.js` 里有完全错误的 PEOPLE 列表

**位置：** `components.js` 第 11 行

**现象：** 代码里有一份与博维团队毫无关系的人员列表：
```js
const PEOPLE = ["冯海林", "袁广民", "关启贝", "肖力", "杨超", "马鑫", "曹新一", "龙小铭"];
```
"袁广民"、"关启贝"等人不是博维团队成员，是其他项目的遗留测试数据。

**风险：** 目前此变量没被引用，但任何人基于它开发新功能都会引入错误数据。

**修复方向：** 清理掉该变量，或替换为真实团队成员列表并从动态 API 加载。

## 二、功能假入口（按钮存在但点了没用）

### 假-01 语音录音是假的

**位置：** `app.js` → `toggleRecording()`

**现象：** 点击录音按钮只是启动了一个秒表，没有实际录音能力。

**根因：** 没有调用 `navigator.mediaDevices.getUserMedia()` 和 `MediaRecorder` API。

**修复方向：** 实现真实的浏览器录音，录制完成后将 Blob 上传到服务器或调用语音转文字 API。

---

### 假-02 AI 提取始终返回固定 mock 数据

**位置：** `app.js` → `extractUpdate()` → `mockSuggestion()`

**现象：** `/api/updates/extract` 接口调用失败时（目前大概率失败），fallback 的 `mockSuggestion()` 永远返回"顾问作业AI化"的固定数据，与用户实际输入内容完全无关。

**根因：**
```js
try {
  suggestion = (await api("/api/updates/extract", { ... })).suggestion;
} catch (error) {
  suggestion = mockSuggestion(payload);  // 永远是顾问作业AI化
}
```

**修复方向：** 接入真实的 AI 接口（Claude API / 豆包），去掉固定 mock fallback，或至少让 mock 根据输入文本做简单关键词匹配。

---

### 假-03 "升级为问题/决策项"不起作用

**位置：** `app.js` → `upgradeTaskIssue()`

**现象：** 点击后只是 toast 一句话并跳转到问题页，没有预填任何数据，用户需要从零开始手填。

**根因：**
```js
function upgradeTaskIssue(event, taskId) {
  toast(`已定位任务 ${taskId}，请在问题与决策页确认升级类型。`);
  switchPage("issues");
}
```

**修复方向：** 跳转时携带 `taskId` 和任务基本信息，在问题页自动打开预填了任务描述、负责人、专项的新建问题表单。

---

### 假-04 成果库"待上传/查看"按钮不起作用

**位置：** `app.js` → `uploadAssetLink()`

**现象：** 点击只弹出 toast，没有任何输入框，无法上传或填写文件链接。

**根因：**
```js
function uploadAssetLink(id) {
  toast(`成果 #${id} 当前只存文件链接，请在后续详情中补充链接地址。`);
}
```

**修复方向：** 打开 Modal，提供一个文本输入框让用户粘贴链接地址，保存后调用 `PUT /api/achievements/{id}` 更新 `file_link` 字段。

---

### 假-05 专项进度卡片"进入专项"不带上下文

**位置：** `app.js` → `projectCard()` 里的按钮

**现象：** 知识资产AI化和顾问作业AI化的"进入专项"按钮跳转到完全一样的工作推进表，用户还需要再手动选一次专项筛选。

**根因：**
```js
<button onclick="switchPage('tasks')">进入专项</button>
```

**修复方向：**
```js
<button onclick="switchPage('tasks'); setTaskProject('${esc(project)}')">进入专项</button>
```

---

### 假-06 驾驶舱告警按钮跳转不带筛选

**位置：** `app.js` → `renderDashboard()` 里的告警栏

**现象：** "延期 5"按钮跳到工作推进表但不筛选延期任务，"决策 3"跳到问题页但不筛选待决策状态。

**修复方向：**
```js
// 延期按钮
onclick="state.taskFilters.status='延期'; switchPage('tasks')"
// 决策按钮
onclick="state.issueTab='decision'; switchPage('issues')"
```

---

### 假-07 "新增问题/决策项"是 toast，不是表单

**位置：** `app.js` → `createIssueDraft()`

**现象：** 点击后只显示"请从提交进度更新页写入"的提示，没有直接创建问题的表单入口。

**修复方向：** 打开一个包含必要字段（描述、专项、负责人、优先级、类型）的 Modal 表单，提交后调用 `POST /api/issues`。

---

### 假-08 会议纪要"完成校对"不写数据库

**位置：** `app.js` → `markMeetingReviewed()`

**现象：** 只是前端解禁"发布"按钮，校对人和校对时间没有记录到数据库。

**修复方向：** 调用 API 记录校对状态，如 `PATCH /api/updates/{id}` 打上"已校对"标记。

---

### 假-09 `jumpToTask()` 跳转不定位

**位置：** `app.js` → `jumpToTask()`

**现象：** 点成果里的"任务#xx"只是跳到工作推进表，不滚动也不展开具体任务。

**修复方向：** 跳转后通过 `data-task-id` 定位元素，滚动到该任务卡片并自动展开 `<details>`。

## 三、数据与状态一致性问题

### 一致-01 `badge()` 的"进行中"颜色是灰色

**位置：** `components.js` → `badge()`

**现象：** 数据库中状态为"进行中"的任务（区别于"推进中"）徽章显示为灰色，看起来像未开始。

**根因：**
```js
if (["推进中", "待确认", "处理中"].includes(text)) tone = "blue";
// "进行中" 不在列表里，显示默认灰色
```

**修复：**
```js
if (["推进中", "进行中", "待确认", "处理中"].includes(text)) tone = "blue";
```

---

### 一致-02 `components.js` 和 `app.js` 都定义了 PROJECTS，语义分裂

**位置：** `components.js` 第1行 vs `app.js` 的 `Object.defineProperty`

**现象：** `components.js` 里有静态死值 `const PROJECTS = [...]`，`app.js` 里又定义了动态 getter。两个同名变量作用域不同（`const` 不挂载到 `window`），将来维护者很容易混淆。

**修复方向：** 删掉 `components.js` 里的静态 `PROJECTS` 声明，只保留 `app.js` 里的动态 getter。

---

### 一致-03 角色切换后 `confirmationTab` 不重置

**位置：** `app.js` → `initRoleSwitch()` 的 change 事件

**现象：** 袁金玉操作"待分配"队列后，切换到一个没有分配权限的负责人，Tab 还停在"待分配"，显示内容与该角色的权限不匹配。

**修复方向：**
```js
node.addEventListener("change", () => {
  state.confirmationTab = "待确认";  // 加这一行
  state.selectedConfirmationId = null;
  // ...
});
```

## 四、体验残缺

### 体验-01 `rejectConfirmation()` 用了浏览器原生 `prompt()`

**位置：** `app.js` → `rejectConfirmation()`

**现象：** 退回操作弹出系统原生对话框，样式无法控制，风格与整体界面完全割裂。

**修复方向：** 改用自定义 Modal，包含一个 `<textarea>` 让用户填写退回原因。

---

### 体验-02 几乎所有 API 调用没有错误处理

**位置：** 全局

**现象：** 除 `extractUpdate()` 和 `onTaskToggle()`，其余所有 `api()` 调用都没有 `try/catch`。接口 500 或网络中断时，页面直接崩溃或静默失败，用户看不到任何提示。

**修复方向：** 对每个 `render*()` 函数加 try/catch，失败时展示错误提示而不是白屏；至少对写操作（POST/PUT/PATCH/DELETE）的调用点加错误 toast。

---

### 体验-03 没有任何加载状态

**位置：** 全局

**现象：** 切换页面时 API 正在请求，页面要么空白，要么显示旧内容，没有任何 loading 指示，用户不知道系统是否在响应。

**修复方向：** 每个 `render*()` 函数开始时在目标容器里显示骨架屏或"加载中..."占位，数据到达后再替换。

---

### 体验-04 没有直接编辑任务的入口

**位置：** `app.js` → `taskCard()`

**现象：** 任务卡片展开只能查看，不能直接修改。想改一个任务字段必须走完整的"提交→AI提取→确认"流程。

**修复方向：** 在展开区域加"编辑"按钮，打开 Modal 直接 `PUT /api/tasks/{id}`，同时写入操作日志。

---

### 体验-05 没有删除功能

**位置：** 全局

**现象：** 后端有 `DELETE /api/tasks/{id}`、`DELETE /api/issues/{id}`、`DELETE /api/achievements/{id}` 接口，前端完全没有删除入口。录错的数据无法从界面清理。

**修复方向：** 在编辑 Modal 里加"删除"按钮（加权限控制，仅 CEO / tech_admin 可见），二次确认后调用 DELETE 接口。

---

### 体验-06 成果库排序方式无说明

**位置：** `app.js` → `sortByAchievementTime()`

**现象：** 成果按 `updated_at DESC` 排序，但页面上没有任何提示，用户看到成果顺序在变化会困惑。

**修复方向：** 在表头或筛选栏旁注明"按最近更新排序"，或提供排序切换选项。
