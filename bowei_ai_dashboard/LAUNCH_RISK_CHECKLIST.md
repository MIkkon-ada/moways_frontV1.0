## 3. P0 风险

### P0-01：多 worker + 内存 session

- **现状**：登录态仍依赖内存 session，若生产进程多 worker，session 可能在不同 worker 间丢失
- **后果**：用户登录后随机失效、退出、权限异常
- **结论**：阻塞正式上线
- **建议**：改为可共享的 session 方案，或统一单 worker 并给出明确生产部署策略

### P0-02：前端生产接线未闭合

- **现状**：前端已独立为 `frontend/` 项目，但生产静态资源、SPA fallback、API 接线尚未形成完整上线方案
- **后果**：正式环境可能出现刷新 404、静态资源找不到、接口跨域失败
- **结论**：阻塞正式上线
- **建议**：补齐前端部署、Nginx fallback、API base / 同域方案

### P0-03：默认凭据 / 文件型 secret 风险

- **现状**：密码、LLM 配置等仍以文件方式保存在仓库根目录或本地文件中，生产 secret 管理不完整
- **后果**：凭据泄露、误提交、误部署
- **结论**：阻塞正式上线
- **建议**：迁移到正式 secret 管理方案，至少在上线前完成轮换与隔离

### P0-04：后端仓库 baseline 必须保持干净

- **现状**：后端仓库已完成部分拆分提交，但仍需确保最终上线前工作区干净、提交边界清晰
- **后果**：无法可靠回滚、无法审计
- **结论**：阻塞正式上线前的发布冻结
- **建议**：继续完成 L2-0C / L2-1 前的收口和基线整理

## 4. P1 风险

### P1-01：SQLite + 手工迁移

- **现状**：默认仍可使用 SQLite，本地开发可接受，但生产化仍未定稿
- **后果**：生产发布和回滚复杂
- **建议**：完成 L3 决策，明确 SQLite 还是 PostgreSQL

### P1-02：`project_id=NULL` 残留数据治理

- **现状**：历史数据仍可能存在 `project_id=NULL`，需要迁移脚本和人工核查配合治理
- **后果**：统计偏差、确认中心与历史数据兼容复杂
- **建议**：按迁移脚本逐步回填，回填后收口旧兜底路径

### P1-03：无健康检查 / 运维观测

- **现状**：未形成完整健康检查接口、日志轮转、监控告警方案
- **后果**：上线后故障难发现、难定位
- **建议**：在 L4 中补齐

### P1-04：无 Docker / compose / CI/CD

- **现状**：当前没有完整的容器化和持续交付链路
- **后果**：发布过程难复现、难回滚
- **建议**：在 L4 中建立最小可复现部署链路

### P1-05：前端无 E2E

- **现状**：前端构建与人工验收已较充分，但缺少稳定的端到端自动化
- **后果**：回归风险高
- **建议**：在 L6 之前补一条稳定 E2E 或关键路径自动化

## 5. L1-L7 分阶段计划

### L1：前端上线收口

- 目标：完成前端部署接线和 SPA fallback
- 验收：前端静态资源可部署，刷新不 404，接口可稳定调用

### L2：后端生产化

- 目标：处理 session、secret、Cookie、安全、健康检查
- 验收：后端可在生产环境稳定运行，权限与登录不再依赖内存状态

### L3：数据库与迁移

- 目标：明确数据库方案，完成迁移、回填、备份、回滚
- 验收：迁移脚本顺序明确，`project_id=NULL` 残留清理方案明确

### L4：部署与运维

- 目标：补 Docker、Nginx、HTTPS、日志轮转、健康检查、版本标记
- 验收：可重复部署，可回滚，可观测

### L5：安全与权限复测

- 目标：补默认账号轮换、权限矩阵复测、越权接口测试、Cookie / CORS / CSRF 检查
- 验收：关键权限用例全部通过

### L6：UAT 试运行

- 目标：使用真实项目和真实成员做试运行
- 验收：核心流程稳定，反馈闭环建立

### L7：正式上线

- 目标：版本冻结、数据备份、监控上线、回滚预案
- 验收：正式发布可追溯、可回退、可监控
